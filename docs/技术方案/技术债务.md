# 技术债务

本文档记录当前阶段暂不实现、但后续需要重点处理的技术债务。

技术债务不是缺陷，而是为了先跑通 Agent Loop 闭环而主动延后的工程能力。

---

## 一、Runtime Hook 外部队列化和增强

### 背景

Agent 主链路中会产生很多副作用：

- 写入 `conversation_record`。
- 写入 `agent_run_step`。
- 写入 `tool_call`。
- 更新事件 summary。
- 写本地日志。
- 后续同步日志分析系统。

这些动作不应该阻塞 Agent 的核心运行。

如果数据库或日志系统抖动，不能直接影响用户拿到回答。

### 当前取舍

第一版实现进程内轻量 Runtime Hook。

第一版 Hook 负责接收基础运行事件：

```text
UserMessageReceived
LoopStepCompleted
ToolCallCompleted
AssistantMessageGenerated
SessionStateChanged
```

第一版不引入外部队列，也不实现复杂插件系统。

Hook 处理失败时先记录错误，保证不影响 Agent 主链路。

### 后续方案

将进程内 Runtime Hook 演进为更完整的 Runtime Event Bus。

Runtime 仍只负责产生事件：

```text
UserMessageReceived
LoopStepCompleted
ToolCallCompleted
AssistantMessageGenerated
SessionStateChanged
```

Event Bus / Worker 负责异步处理副作用：

```text
写 conversation_record
写 agent_run_step
写 tool_call
更新事件 summary
写 logs
同步外部日志系统
```

### 设计原则

- Hook 失败不能影响 Agent 主链路。
- Hook 处理应支持重试。
- Hook 处理应支持降级，例如先落本地失败队列。
- Hook 不参与 Agent Loop 决策。
- Hook 不修改当前轮 Action 判断结果。

### 可选演进

第一版是进程内轻量 Hook。

后续可替换为：

- Redis Stream。
- Celery。
- Dramatiq。
- Kafka。
- 其他消息队列。

## 二、agent_run_step 日志存储演进

### 背景

`agent_run_step` 是高频追加记录。

它记录每轮 Agent Loop 的状态流转：

```text
state_before → action → state_after
```

第一版放在 PostgreSQL 中，方便开发、调试和闭环验证。

但随着请求量、worker 调用量、Skill 调用量增加，`agent_run_step` 可能成为高频写入表。

### 当前取舍

第一版使用 PostgreSQL。

原因：

- 工程复杂度低。
- 方便和 `agent_run` 关联排查。
- 不引入额外中间件。
- 有利于先跑通 Loop 闭环。

### 后续方案

当日志量变大后，将 `agent_run_step` 演进到日志或分析中间件。

可选方向：

- Elasticsearch。
- OpenSearch。
- ClickHouse。
- Kafka + 消费者落库。

### 推荐演进路径

```text
日志演进第一步：
Runtime → PostgreSQL agent_run_step

日志演进第二步：
Runtime → PostgreSQL agent_run_step
        → 异步同步到 ES / OpenSearch

日志演进第三步：
Runtime → Event Queue
        → PostgreSQL 保留必要记录
        → ES / OpenSearch / ClickHouse 保存完整状态流转日志
```

### 设计原则

- `agent_run_step` 保持 append-only。
- 不更新历史 step。
- 主流程不依赖 ES / OpenSearch 才能运行。
- PostgreSQL 仍保存必要的可恢复数据。
- 日志中间件用于检索、排查、分析和可观测性。

## 三、记忆宫殿服务接入

### 背景

记忆宫殿会作为独立服务或独立 agentic RAG 项目存在。

当前 Agent 框架不直接实现记忆宫殿的存储、管理和检索算法。

Context Engineer 只需要在上下文组装阶段预留内置 Provider 调用入口。

第一版不实现记忆管理。

第四阶段先将 `memory_manager` 作为普通 worker 子 Agent 接入，由 main 通过 `call_agent` 调用。

Hook 触发后台 `memory_manager` 的异步模式作为后续演进。

### 当前取舍

第一版只实现内置 `MemoryPalaceProvider` 适配器入口。

如果外部记忆宫殿服务尚未完成，适配器允许返回空结果。

空结果不应影响 Agent Loop 主链路。

第三阶段 Worker Task Package 只预留以下字段：

```text
retrieved_context
long_term_memory
memory_palace_refs
```

第三阶段不实现 LLM 对话压缩、不实现事件 summary 检索、不实现长期记忆召回、不实现记忆宫殿召回。

第四阶段需要把这些字段完整接入 main → worker 的上下文传递链路。

记忆宫殿不是行为准则存储。

它主要承载：

- 用户提到的重要事件。
- 项目事实和项目背景。
- 长期可召回的上下文线索。
- 对话中形成的关键共识。
- 和当前任务未来可能相关的经验片段。

它不承载：

- Agent 的行为准则。
- Agent 的输出格式规则。
- Agent 的工具调用规则。
- 应写入 `Instruction.md`、`output.md` 或长期设定表的稳定偏好。

技术方案中只体现：

```text
Context Engineer
  ↓
MemoryPalaceProvider
  ↓
Memory Palace Retrieval
  ↓
注入 Prompt Context
```

### 后续方案

后续需要补充：

- 将 `memory_manager` 从同步 worker 调用演进为 Runtime Hook 触发的后台任务。
- Runtime Hook 触发 `MemoryReflectionRequested` 事件。
- 后台 `memory_manager` 子 Agent 的职责和输出结构。
- `memory_manager` 判断写入长期记忆、记忆宫殿或忽略的规则。
- 记忆宫殿服务地址和认证方式。
- 检索请求参数。
- 返回结果结构。
- 写入请求结构。
- 写入候选的来源标记，例如用户主动要求记录、Agent 从对话中推断值得沉淀。
- namespace / user_id / agent_id 的作用域规则。
- 结果排序和截断策略。
- 失败降级策略。
- 管理接口，例如写入、更新、删除和归档。

### 设计原则

- 记忆宫殿服务不可用时，不阻断 Agent 主链路。
- Context Engineer 只消费检索结果，不负责管理记忆宫殿内部结构。
- 记忆宫殿是 Context Engineer 的内置上下文检索本能，不建模为普通 Skill。
- 第四阶段先由普通 worker `memory_manager` 判断和编排记忆管理。
- 后续再演进为 Runtime Hook 异步触发的后台 `memory_manager`。
- 记忆宫殿的底层写入通过外部服务接口承接。
- 第一版不把记忆宫殿内容落到本项目数据库。

## 四、事件级 Summary 上下文演进

### 背景

主 Agent 和 worker Agent 之间需要共享“当前这件事”的上下文。

如果只依赖当前用户输入和最近一次 `previous_action_result`，worker 在多轮补充信息时容易丢失前序任务意图，例如第一轮用户要求“路线 + 天气”，第二轮只补充“从 A 到 B，开车”，worker 可能只看到路线，而丢掉天气。

因此后续需要引入事件级 summary 上下文：

```text
单轮或多轮用户交互
  ↓
归属同一个 agent_event
  ↓
main run / worker run / skill call 绑定 event_id
  ↓
Context Engineer 按 event_id 取事件上下文
```

worker Agent 不创建独立事件。

当 main Agent 调用 worker 时，worker run 使用 main 当前事件的 `event_id`。worker 自己的 `agent_run`、`agent_run_step` 和 `tool_call` 都归属这个事件。

这样 worker 的对话上下文、执行历史和 Skill 调用结果可以通过同一个 `event_id` 获取，避免 worker 只看到当前 handoff，而忘记同一事件内前面已经执行到哪里。

### 当前取舍

第一版不做跨事件检索，也不让模型判断事件归属。

事件上下文只基于当前 `event_id` 获取。

main 和 worker 使用同一个 `event_id`，但按 `agent_id` 分开维护 summary：

```text
agent_event
  └── agent_event_context(event_id, agent_id = main)
  └── agent_event_context(event_id, agent_id = amap_worker)
```

事件 summary 生成策略：

```text
主 Agent Harness 或主 Agent 最终状态认为事件结束
  ↓
Runtime 按 event_id + agent_id 汇总该 agent 的对话和执行历史
  ↓
未超过轮数 / 字数限制：直接拼接汇总为 summary
  ↓
超过限制：调用模型总结为约 100 字 summary
  ↓
写入 agent_event_context.summary
```

建议第一版限制：

```text
EVENT_SUMMARY_DIRECT_MAX_TURNS=6
EVENT_SUMMARY_DIRECT_MAX_CHARS=1200
EVENT_SUMMARY_TARGET_CHARS=100
```

### 后续方案

未来可以把事件 summary 交给更完整的记忆服务或 `memory_manager` 管理。

后续可演进为：

```text
agent_event_context.summary
  ↓
memory_manager / memory service 组织事件摘要
  ↓
基于 summary 做跨事件检索
  ↓
由模型或记忆服务参与更精细的事件关系判断
```

也就是说，第一版只沉淀当前事件 summary，不做跨事件检索。

后续再考虑让模型或记忆服务基于 summary 检索旧事件。

### 设计原则

- 事件状态不是 Loop 状态。
- `agent_event.status` 由 Runtime 或主 Harness 结果推动，不由 worker 自行管理。
- worker 使用 main 当前 `event_id`，不另建事件。
- worker 的上下文也基于 `event_id` 获取同一事件下的 main / worker 运行历史。
- main 和 worker 的 summary 按 `agent_id` 分开保存。
- worker 不直接查事件表，仍由 Runtime / Context Engineer 拼接上下文。
- conversation_record 保留原始对话，不再承担检索内容。
- agent_event_context 承担事件级 summary 上下文。

## 五、Agent 辅助新增 Skill

### 背景

未来业务模块可能希望主 Agent 根据业务需求辅助创建新的 Skill。

例如：

- 用户发现当前 Agent 缺少某类稳定能力。
- 管理员希望把一段重复流程沉淀为 Skill。
- Agent 在长期运行中发现某类任务经常出现，可以建议沉淀为 Skill。

这个能力有价值，但不能让主 Agent 直接自由写文件、执行代码或自动上线 Skill。

### 当前取舍

第二阶段不实现 Agent 自动新增 Skill。

当前只实现：

```text
已有 Skill 能被发现、校验、执行、评估、记录。
```

Skill 的新增、修改、删除仍然由开发者或管理员通过工程流程完成。

### 后续方案

未来可以设计成 Skill 管理闭环：

```text
用户 / 管理员提出新增 Skill
  ↓
main agent 判断是否适合沉淀为 Skill
  ↓
生成 SkillProposal
  ↓
skill_manager worker 或管理接口审查
  ↓
生成 SKILL.md 草案
  ↓
生成 schema.json 草案
  ↓
选择 adapter 类型
  ↓
校验命名、schema、安全和冲突
  ↓
人工确认或审批
  ↓
写入 agents/main/skills/<skill_id>/
  ↓
Runtime 重新加载 SkillRegistry
```

建议拆成四个阶段性对象：

```text
SkillProposal
SkillReview
SkillInstall
SkillRegistryReload
```

主 Agent 可以参与：

- 识别是否需要新 Skill。
- 设计 Skill 用途、适用场景和不适用场景。
- 生成 `SKILL.md` 草案。
- 生成 `schema.json` 草案。
- 生成调用示例。
- 建议 adapter 类型：`http / chain / code`。

### 按 Skill 类型的落点规则

未来 Agent 辅助新增 Skill 时，必须先判断 Skill 类型，再决定改哪些位置。

#### API Skill

API Skill 只新增 Skill 资产，不新增 Runtime 代码。

落点：

```text
agents/main/skills/<skill_id>/
  SKILL.md
  schema.json
```

`schema.json` 中声明：

```json
{
  "executor": {
    "type": "api",
    "endpoint": "https://example.com/api",
    "method": "POST",
    "auth": {
      "type": "query",
      "param": "key",
      "env": "SOME_API_KEY"
    },
    "timeout_ms": 10000
  },
  "input_schema": {},
  "output_schema": {},
  "response_mapping": {}
}
```

规则：

- API 地址、鉴权方式、超时和响应映射写在 `schema.json`。
- 密钥只写 `.env` 或服务端配置，不写入 Skill 资产。
- 不修改 `runtime/tools/builtin_skills/`。
- 不新增 Python executor。
- 需要校验 `response_mapping` 是否能转换为统一 SkillResult。

#### Code Skill

Code Skill 需要新增 Skill 资产，并新增受控 Runtime 代码。

落点：

```text
agents/main/skills/<skill_id>/
  SKILL.md
  schema.json

runtime/tools/builtin_skills/<executor_name>.py
runtime/tools/builtin_skills/__init__.py
```

还需要更新白名单注册：

```text
CODE_SKILL_EXECUTORS
```

`schema.json` 中声明：

```json
{
  "executor": {
    "type": "code",
    "name": "<executor_name>"
  },
  "input_schema": {},
  "output_schema": {}
}
```

规则：

- 不允许从 Skill 目录动态 import 任意代码。
- 只允许调用 Runtime 白名单注册过的 executor。
- 新增或修改代码必须经过更严格审批。
- executor 必须返回统一 SkillResult。
- executor 不能直接返回用户内容，结果仍需进入 Harness。

#### Chain Skill

Chain Skill 需要新增 Skill 资产，并新增或更新受控链路注册。

落点：

```text
agents/main/skills/<skill_id>/
  SKILL.md
  schema.json

runtime/tools/chains/<chain_id>.py
runtime/tools/chains/__init__.py
```

还需要更新链路白名单注册：

```text
CHAIN_SKILL_EXECUTORS
```

`schema.json` 中声明：

```json
{
  "executor": {
    "type": "chain",
    "chain_id": "<chain_id>"
  },
  "input_schema": {},
  "output_schema": {}
}
```

规则：

- Chain 的步骤必须来自 Runtime 已注册 Skill 或已注册链路能力。
- 不允许用户输入直接动态生成任意执行步骤。
- 每一步都必须产生结构化中间结果。
- 最终必须返回统一 SkillResult。
- Chain Skill 的新增或修改需要审批。

### 安装流程约束

未来 Agent 可以生成草案，但安装动作必须由受控流程完成。

建议流程：

```text
SkillProposal
  ↓
类型判断：api / code / chain
  ↓
生成资产草案
  ↓
如需代码或链路，生成 Runtime 变更草案
  ↓
schema 校验
  ↓
安全校验
  ↓
命名冲突校验
  ↓
人工确认或管理员审批
  ↓
写入文件
  ↓
更新白名单注册
  ↓
SkillRegistry reload
  ↓
测试调用
```

主 Agent 不应直接执行：

- 任意写代码。
- 任意改 Runtime。
- 任意调用外部接口。
- 自动上线 Skill。
- 自动覆盖已有 Skill。
- 绕过人工确认或管理接口。

### 设计原则

- 新增 Skill 是管理流程，不是普通用户对话副作用。
- Skill 安装必须经过校验和确认。
- Skill 目录写入应由受控管理接口或 `skill_manager` worker 完成。
- `skill_id` 必须全局唯一，且不可静默覆盖。
- 新增 Skill 后需要触发 SkillRegistry reload。
- reload 失败不能影响正在执行的 Agent Run。
- Skill 创建能力建议放到第五阶段或更后，不进入第二阶段 Skill 闭环。
