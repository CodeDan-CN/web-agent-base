# Agent 脚手架设计

## 设计定位

这个脚手架是一个单模块 Agent 工程模板。

它不负责在一个工程里管理多个业务模块，也不负责做平台级多模块路由。

更准确的使用方式是：

```text
一个业务模块
  ↓
拉取一份 Agent 脚手架
  ↓
编写一个 main agent 和多个 worker agent
  ↓
构建 Docker
  ↓
部署为该模块的 Agent 服务
```

如果另一个业务模块也需要 Agent 能力，就重新拉取一份脚手架，独立编写、独立构建、独立部署。

因此这个脚手架只考虑：

1. 单模块。
2. 单个主 Agent。
3. 多个 worker Agent。
4. 模块内多用户使用。
5. 所有用户共享同一套 Agent MD 文件。
6. 用户差异只来自长期记忆、短期状态、记忆宫殿检索结果和当前请求。

简单来说：

```text
脚手架是单个业务模块的 Agent 工程，不是多模块 Agent 平台。
```

## 总体目录结构

建议项目采用下面的结构：

```text
agent-base/
├── .env                          # 本地环境变量，保存数据库、模型、RAG 服务等连接配置
├── Agent.md                      # 当前模块 Agent 服务的整体说明
├── app.py                        # FastAPI 应用入口，暴露当前模块的 Agent API
├── requirements.txt              # conda 环境内 pip 依赖清单
├── docs/                         # 项目设计文档
│   ├── Agent Prompt整体构成.md     # 说明完整 Agent Prompt 的组成
│   ├── Agent人设管理.md            # 说明 SOUL / Agent / Instruction 的职责
│   ├── Agent记忆管理.md            # 说明长期记忆、记忆宫殿等设计
│   ├── Loop设计.md                 # 说明 Agent Loop / Runtime / Harness 流程
│   ├── 技术债务.md                 # 记录第一版暂缓实现但后续要处理的工程问题
│   └── 脚手架设计.md               # 当前脚手架目录设计说明
│
├── web/                          # FastAPI 接口层
│   └── ...                       # 路由、请求响应结构、依赖注入等接口层内容
│
├── schema/                       # 数据库实体和接口参数实体
│   ├── db/                       # 数据库实体、ORM Model、表结构映射
│   └── api/                      # 请求参数、响应参数、接口 DTO
│
├── agents/                       # 当前模块的 Agent 角色资产
│   ├── main/                     # 当前模块唯一入口主 Agent
│   │   ├── SOUL.md               # Agent 身份、气质、表达风格
│   │   ├── Agent.md              # Agent 目的、职责、边界
│   │   ├── Instruction.md        # Agent 在不同场景下的 SOP
│   │   ├── tools.md              # 可用 Skill、worker 和工具说明
│   │   ├── output.md             # 最终回答的输出规范
│   │   ├── harness.md            # Skill / worker 结果评估规则
│   │   ├── skills/               # main 私有或局部绑定的 Skill
│   │   │   └── one_skill/        # 示例 Skill：某项能力
│   │   │       ├── SKILL.md      # Skill 入口说明和元信息
│   │   │       ├── schema.json   # Skill 输入 / 输出结构约束
│   │   │       ├── references/   # 可选，Skill 使用参考资料
│   │   │       └── assets/       # 可选，Skill 模板或静态资源
│   │   └── prompts/              # main 专属 Prompt 差异片段
│   │       ├── fragments.md      # main 特有提示词片段
│   │       └── examples.md       # main 少量 few-shot 示例
│   │
│   └── workers/                  # 被 main 调用的 worker Agent 集合
│       ├── researcher/           # 示例 worker：负责检索和分析
│       │   ├── SOUL.md           # worker 身份和工作气质
│       │   ├── Agent.md          # worker 职责和边界
│       │   ├── Instruction.md    # worker 执行 SOP
│       │   ├── tools.md          # worker 可用工具说明
│       │   ├── output.md         # worker 返回结果的格式规范
│       │   ├── harness.md        # worker 内部结果评估规则
│       │   ├── skills/           # researcher 私有或局部绑定的 Skill
│       │   │   └── one_skill/    # 示例 Skill：某项能力
│       │   │       ├── SKILL.md  # Skill 入口说明和元信息
│       │   │       └── schema.json # Skill 输入 / 输出结构约束
│       │   └── prompts/          # researcher 专属 Prompt 差异片段
│       │       ├── fragments.md  # researcher 特有提示词片段
│       │       └── examples.md   # researcher 少量 few-shot 示例
│       │
│       └── planner/              # 示例 worker：负责规划和拆解
│           ├── SOUL.md           # worker 身份和工作气质
│           ├── Agent.md          # worker 职责和边界
│           ├── Instruction.md    # worker 执行 SOP
│           ├── tools.md          # worker 可用工具说明
│           ├── output.md         # worker 返回结果的格式规范
│           ├── harness.md        # worker 内部结果评估规则
│           ├── skills/           # planner 私有或局部绑定的 Skill
│           └── prompts/          # planner 专属 Prompt 差异片段
│
├── runtime/                      # Agent 执行系统
│   ├── agents/                   # Agent 加载、解析、上下文构建
│   ├── loop/                     # Agent Loop 判断下一步 Action
│   ├── state/                    # State、pending_action、loop step 管理
│   ├── harness/                  # Skill / worker 执行结果评估
│   ├── context/                  # 长期记忆、短期状态、RAG 结果装配
│   └── tools/                    # Skill、工具、worker 调用适配
│
├── utils/                        # 公共工具方法
│   └── ...                       # 通用函数、格式转换、时间处理等
│
├── exception/                    # 全局异常和自定义异常
│   └── ...                       # 业务异常、Runtime 异常、异常处理器等
│
├── deployment/                   # 部署相关文件
│   └── ...                       # Docker、启动脚本、部署说明等
│
└── logs/                         # 本地文件日志
    ├── runtime.log               # Runtime 执行日志
    ├── tools.log                 # Skill / 工具调用日志
    ├── workers.log               # worker 调用日志
    └── error.log                 # 错误日志
```

## 目录说明

这一版文档只说明脚手架结构和每个大目录的职责。

具体 Runtime 流程、Prompt 拼装、数据库表设计、记忆服务接口、API 入参出参等技术细节，放到下一步技术方案里展开。

### 根目录文件

根目录文件是当前模块 Agent 服务的启动入口和基础运行配置。

| 文件 | 作用 |
| --- | --- |
| `.env` | 保存当前部署环境相关的敏感配置，例如数据库连接、模型 Key、RAG 服务地址等 |
| `Agent.md` | 当前模块 Agent 服务的整体说明，描述模块目标、服务边界和整体能力 |
| `app.py` | FastAPI 应用入口，只负责创建 app、挂载 web 路由、初始化基础组件 |
| `requirements.txt` | conda 环境内需要安装的 pip 依赖清单 |

根目录不放 Agent 人设、Prompt、记忆内容或复杂业务逻辑。

### web/

`web/` 是当前模块的接口层。

它负责把外部 HTTP 请求接进来，然后转交给 Runtime 和 main agent。

`web/` 不负责 Agent 推理逻辑，也不直接编排 worker。

它主要负责：

- 定义 FastAPI 路由。
- 定义请求和响应结构。
- 校验接口入参。
- 注入 Runtime、配置和数据库连接等依赖。
- 调用 Runtime 执行当前模块的 main agent。
- 将 Runtime 输出转换成 HTTP 响应。

建议边界是：

```text
HTTP Request
  ↓
web/
  ↓
runtime
  ↓
agents/main
```

这样 `app.py` 只做应用入口，`web/` 负责接口层，`runtime/` 负责 Agent 执行。

### schema/

`schema/` 保存结构化数据实体。

它主要分成两类：

- `schema/db/`：数据库实体、ORM Model、表结构映射。
- `schema/api/`：接口请求参数、响应参数、DTO 等。

`schema/` 只放数据结构定义，不写业务流程，不执行 Agent 逻辑。

这样做可以让 `web/`、`runtime/`、数据库访问层复用同一套实体定义。

### docs/

`docs/` 保存当前 Agent 框架的设计文档。

它的作用是记录设计共识，帮助后续实现时保持方向一致。

这里主要放：

- Agent Prompt 的整体构成。
- Agent 人设和行为文件的设计。
- Agent 记忆系统的设计。
- Agent Loop / Runtime / Harness 的流程设计。
- 当前脚手架目录结构说明。

`docs/` 不参与 Runtime 执行。

### agents/

`agents/` 保存当前模块的 Agent 角色资产。

它是这个脚手架最核心的业务目录。

当前约定一个模块只有一个 `main` agent，另有多个 `worker` agent。

```text
agents/
  main/
  workers/
```

`main` 是当前模块唯一入口 Agent，负责理解用户请求、决定下一步动作、调度 worker，并最终对用户输出结果。

`workers` 是被 `main` 调用的辅助 Agent 集合，每个 worker 负责一个相对独立的局部能力，例如检索、规划、写作、审核等。

所有用户共享同一套 `agents/` 文件。

用户差异不写进 Agent 文件，而是通过长期记忆、短期状态、记忆宫殿检索结果和当前请求进入上下文。

每个 Agent 都可以拥有自己的 `skills/` 目录。

`tools.md` 和 `skills/` 的区别是：

- `tools.md`：给 Agent 和 Runtime 看的能力说明，描述可用 Skill、worker、工具边界和适用场景。
- `skills/`：放当前 Agent 私有或局部绑定的 Skill 资产，例如 Skill 配置、描述文件、适配说明等。

如果某个 Skill 是整个模块通用能力，可以由 main 和多个 worker 共同声明使用。

如果某个 Skill 只服务某个 worker，就放在该 worker 自己的 `skills/` 下。

单个 Skill 建议采用下面的最小结构：

```text
skills/
  skill-name/
    SKILL.md
    schema.json
    references/
    assets/
```

文件含义：

| 路径 | 作用 |
| --- | --- |
| `SKILL.md` | Skill 的入口说明，描述名称、用途、适用场景、调用注意事项 |
| `schema.json` | Skill 的输入 / 输出结构约束，方便 Runtime 校验参数和结果 |
| `references/` | 可选，保存 Skill 需要参考的说明文档、规范或样例 |
| `assets/` | 可选，保存 Skill 使用的模板、静态资源或示例文件 |

这里参考的是 ClawHub / AgentSkills / Codex Skills 中常见的 `SKILL.md` 入口形式，但不会在脚手架阶段设计成完整插件市场。

当前只把 Skill 当成 Agent 局部能力资产。

### agents/main/

`agents/main/` 是当前模块的主 Agent。

它是 Runtime 默认加载的入口。

这里放 main agent 的身份、职责、SOP、工具说明、输出规范、评估规则和少量 Prompt 差异片段。

main agent 的重点不是直接完成所有事情，而是判断：

- 是否可以直接回答用户。
- 是否需要调用 worker。
- 是否需要调用 Skill。
- 是否需要向用户追问。
- 是否已经可以汇总结果并完成回答。

### agents/workers/

`agents/workers/` 保存 worker agent。

worker 不直接面向最终用户，而是被 main agent 调用。

每个 worker 都应该有清晰、收敛的职责。

例如：

- `researcher`：负责检索、阅读、整理信息。
- `planner`：负责拆解任务、制定步骤。
- `writer`：负责生成文本草稿。
- `reviewer`：负责检查结果质量。

worker 的输出会返回给 main，由 main 判断下一步。

### runtime/

`runtime/` 保存 Agent 执行系统。

它不描述某个 Agent 的人设，也不保存某个用户的记忆。

它负责让 `agents/` 中定义的角色真正跑起来。

主要职责包括：

- 加载 main agent 和 worker agent。
- 根据当前 State 判断下一步 Action。
- 调用 Skill 或 worker。
- 管理短期状态和 pending_action。
- 评估 Skill / worker 的执行结果。
- 组装当前轮模型调用所需上下文。

后续技术方案会进一步拆解 `runtime/agents`、`runtime/loop`、`runtime/state`、`runtime/harness`、`runtime/context`、`runtime/tools` 的具体实现。

### utils/

`utils/` 保存公共工具方法。

它适合放跨目录复用的小函数，例如：

- 时间处理。
- 字符串处理。
- ID 生成。
- JSON / dict 转换。
- 通用格式化。
- 简单校验函数。

`utils/` 不应该承载业务主流程，也不应该放 Agent 决策逻辑。

### exception/

`exception/` 保存全局异常和自定义异常。

它适合放：

- 通用业务异常。
- Runtime 异常。
- Skill 调用异常。
- worker 调用异常。
- 参数异常。
- 全局异常处理器。

这样可以让 `web/` 返回统一错误结构，也方便 Runtime 内部区分不同失败类型。

### deployment/

`deployment/` 保存当前模块 Agent 服务的部署交付文件。

它适合放：

- Docker 相关文件。
- docker-compose.yaml。
- 启动脚本。
- 部署说明。
- 环境初始化说明。
- 不同环境的部署模板。

`deployment/` 不放 Agent 人设、Runtime 逻辑或用户数据。

它的职责是帮助这份单模块 Agent 工程被构建、启动和部署。

### logs/

`logs/` 保存本地文件日志。

它主要用于开发调试和排查问题。

结构化业务数据不放在 `logs/`，例如 run 记录、State 变化、工具调用结果等，应该进入外部数据库。

`logs/` 适合保存：

- Runtime 启动和执行日志。
- Skill / 工具调用日志。
- worker 调用日志。
- 错误日志。

生产环境中，`logs/` 也可以被容器日志、云日志或集中式日志系统替代。
