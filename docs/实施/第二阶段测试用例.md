# 第二阶段测试用例

## 概述

本文档定义第二阶段「Skill 闭环」测试用例。

第二阶段测试目标是验证：

- Skill 可以通过 `agents/main/skills/` 扫描加载。
- `SKILL.md` 和 `schema.json` 可以形成 SkillDefinition。
- `SkillAdapter` 支持 `api / chain / code` 三类执行方式。
- `call_skill` 可以使用真实 `skill_id` 调用 Skill。
- Skill 输入和输出都经过 JSON Schema 校验。
- Skill 执行结果进入 LLM Harness。
- Harness 之后仍然回到 Loop，由 `answer_user` 生成最终回答。
- AG-UI 可以输出真实 Skill 的中间步骤。

## 一、预置 Skill

第二阶段预置以下 Skill。

| skill_id | 类型 | 用途 |
| --- | --- | --- |
| `content_extract` | `code` | 文本摘要、关键要点提取、大纲整理 |
| `amap_geocode` | `api` | 地址转经纬度、adcode、citycode |
| `amap_weather` | `api` | 根据 adcode 查询天气 |
| `amap_direction_driving` | `api` | 根据起终点经纬度查询驾车路线 |
| `travel_briefing_formatter` | `code` | 将路线和天气组合成出行建议 |
| `amap_route_weather_plan` | `chain` | 地址解析 → 驾车路线 → 目的地天气 → 出行建议 |

### 1.1 环境变量

高德 Web 服务 Key 只写入 `.env`，不写入 Skill 资产文件，不写入 Prompt。

建议变量名：

```text
AMAP_WEB_SERVICE_KEY=***
```

测试脚本读取该变量后，由 `ApiSkillAdapter` 追加到请求 query 参数中。

### 1.2 高德 API Skill

高德相关 API Skill 使用官方 Web 服务接口。

| Skill | API |
| --- | --- |
| `amap_geocode` | `https://restapi.amap.com/v3/geocode/geo` |
| `amap_weather` | `https://restapi.amap.com/v3/weather/weatherInfo` |
| `amap_direction_driving` | `https://restapi.amap.com/v3/direction/driving` |

对外返回必须统一转换为：

```json
{
  "status": "success",
  "data": {},
  "summary": "执行摘要",
  "missing_params": [],
  "error": null
}
```

## 二、测试原则

第二阶段测试分三层：

1. Skill 资产加载测试。
2. SkillAdapter 执行测试。
3. Agent Loop 全链路测试。

测试必须保持：

- 单轮简单问题使用新 `session_id`。
- 多轮测试每组使用独立新 `session_id`，组内复用同一个 `session_id`。
- 用户消息使用正常表达。
- 用户侧输出不能暴露 schema、Prompt、Adapter、内部异常堆栈。
- 明显缺信息时允许直接 `ask_user → awaiting_user`。
- 用户补充信息后，必须能回到正确 `call_skill` 链路。

---

## 三、统一请求格式

所有 Agent 链路测试通过 `/agent/run` 或 `/agent/run/stream` 触发。

```json
{
  "user_id": "skill_test_user",
  "agent_id": "main",
  "session_id": null,
  "request_id": "case_skill_001",
  "message": "帮我查一下杭州西湖区今天的天气",
  "metadata": {}
}
```

## 五、单 Skill 测试

### 5.1 Code Skill：content_extract 成功

用户输入：

```text
帮我把这段内容提炼成要点：第二阶段要实现 SkillRegistry、SkillAdapter、SkillExecutor、schema 校验和真实 Skill 调用记录。
```

期望链路：

```text
new_request → call_skill(content_extract) → ready_to_plan → answer_user → completed
```

检查点：

- `tool_call.tool_name = content_extract`。
- `execution_result.status = success`。
- Harness 输出 `ready_to_plan`。
- 最终回答包含 Skill 提取结果。

### 5.2 Code Skill：content_extract 缺参

用户输入：

```text
帮我提取一下重点
```

允许链路：

```text
new_request → ask_user → awaiting_user
```

或：

```text
new_request → call_skill(content_extract) → missing_params → ask_user → awaiting_user
```

检查点：

- 如果直接追问，用户侧问题必须明确要求补充原文。
- 如果调用 Skill，Harness 必须判断缺少 `content`。

### 5.3 API Skill：amap_geocode 成功

用户输入：

```text
帮我解析一下地址：杭州市西湖区文三路，城市是杭州
```

期望链路：

```text
new_request → call_skill(amap_geocode) → ready_to_plan → answer_user → completed
```

检查点：

- `ApiSkillAdapter` 被调用。
- 请求参数包含 `address` 和 `city`。
- API Key 只来自 `.env`。
- `tool_call.tool_name = amap_geocode`。
- 输出包含 `location` 和 `adcode`。

### 5.4 API Skill：amap_weather 成功

用户输入：

```text
查询 adcode 330106 的实时天气
```

期望链路：

```text
new_request → call_skill(amap_weather) → ready_to_plan → answer_user → completed
```

检查点：

- `city = 330106`。
- `extensions = base`。
- 输出包含天气、温度、风向、湿度等标准字段。

### 5.5 API Skill：amap_direction_driving 成功

用户输入：

```text
帮我查一下从 120.123456,30.123456 到 120.654321,30.654321 的驾车路线
```

期望链路：

```text
new_request → call_skill(amap_direction_driving) → ready_to_plan → answer_user → completed
```

检查点：

- `origin` 和 `destination` 均为经纬度字符串。
- 输出包含距离、预计时间和路线摘要。

### 5.6 API Skill：缺少 Key

测试方式：

```text
临时移除 AMAP_WEB_SERVICE_KEY 后调用 amap_weather。
```

期望：

```text
RUN_ERROR 或结构化系统错误
```

检查点：

- 不伪装成用户缺参。
- 不向用户暴露内部异常堆栈。
- `agent_run.error_message` 有记录。

## 六、Chain Skill 测试

### 6.1 Chain Skill：地址到路线天气建议

用户输入：

```text
我明天从杭州市西湖区文三路开车去杭州市上城区湖滨银泰，帮我看一下路线和目的地天气，并给出出行建议。
```

期望链路：

```text
new_request → call_skill(amap_route_weather_plan) → ready_to_plan → answer_user → completed
```

链路内部步骤：

```text
1. amap_geocode(origin_address, city)
2. amap_geocode(destination_address, city)
3. amap_direction_driving(origin.location, destination.location)
4. amap_weather(destination.adcode)
5. travel_briefing_formatter(route, weather)
```

检查点：

- `ChainSkillAdapter` 被调用。
- 每个链路步骤都有结构化中间结果。
- 最终 SkillResult 包含 `origin / destination / route / weather / briefing`。
- `tool_call.tool_name = amap_route_weather_plan`。
- 最终回答不能暴露内部链路步骤名，但可以自然说明路线、天气和建议。

### 6.2 Chain Skill：缺少起点

用户输入：

```text
帮我查一下去杭州市上城区湖滨银泰的路线和天气。
```

允许链路：

```text
new_request → ask_user → awaiting_user
```

或：

```text
new_request → call_skill(amap_route_weather_plan) → missing_params → ask_user → awaiting_user
```

检查点：

- 追问应聚焦缺少起点。
- 如果调用 Chain Skill，返回 `missing_params` 包含 `origin_address`。

### 6.3 Chain Skill：地理编码失败

用户输入：

```text
我从一个不存在的火星基地开车去杭州市上城区湖滨银泰，帮我看路线和天气。
```

期望：

```text
call_skill(amap_route_weather_plan) → failed
```

或自然说明地址无法解析。

检查点：

- 失败来自链路中 `amap_geocode`。
- ChainSkillAdapter 返回结构化失败结果。
- 用户侧不暴露接口原始错误。

### 6.4 Chain Skill：多轮补充完成

轮次 1：

```text
帮我规划一下明天开车出行，顺便看天气。
```

允许链路：

```text
new_request → ask_user → awaiting_user
```

轮次 2：

```text
我从杭州市西湖区文三路去杭州市上城区湖滨银泰。
```

期望链路：

```text
awaiting_user → call_skill(amap_route_weather_plan) → ready_to_plan → answer_user → completed
```

检查点：

- 第二轮复用同一个 `session_id`。
- Runtime 可以承接上一轮 pending intent。
- 第二轮进入 Chain Skill。

## 七、AG-UI 流式测试

### 7.1 API Skill 流式事件

请求：

```text
查询 adcode 330106 的实时天气
```

接口：

```text
POST /agent/run/stream
```

检查点：

- 输出 `TOOL_CALL_START`，`toolName = amap_weather`。
- 输出 `TOOL_CALL_ARGS`，但不包含 API Key。
- 输出 `TOOL_CALL_END`，包含状态和摘要。
- 最终回答通过 `TEXT_MESSAGE_CONTENT` 流式输出。

### 7.2 Chain Skill 流式事件

请求：

```text
我从杭州市西湖区文三路开车去杭州市上城区湖滨银泰，帮我看路线和目的地天气。
```

检查点：

- 输出 `TOOL_CALL_START`，`toolName = amap_route_weather_plan`。
- 普通 UI 可以只展示链路摘要。
- 事件中不能输出完整 API Key、原始 Prompt、内部异常堆栈。
- 最终回答通过 `TEXT_MESSAGE_CONTENT` 流式输出。

## 九、验收要求

第二阶段测试通过条件：

- 所有预置 Skill 可以被扫描和加载。
- `executor.type = api / chain / code` 均可执行。
- 所有 Skill 输入都经过 JSON Schema 校验。
- 所有 Skill 输出都经过 JSON Schema 校验。
- 所有 Skill 结果都进入 Harness。
- 所有 `call_skill` 都写入 `tool_call`。
- API Key 不出现在 Prompt、AG-UI 事件和用户回答中。
- Chain Skill 的每一步都有结构化中间结果。
- 多轮补充后可以进入正确 Chain Skill。
- AG-UI 输出真实 `skill_id`。
- 用户侧输出自然，不暴露内部实现细节。
