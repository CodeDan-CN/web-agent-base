# 第三阶段测试用例

## 概述

本文档定义第三阶段「Worker Agent 闭环」测试用例。

第三阶段测试目标是验证：

- main agent 可以扫描和调用真实 worker agent。
- worker agent 可以加载自己的 Agent 资产。
- worker agent 可以加载自己的 SkillCatalog。
- main agent 不直接持有高德领域 Skill。
- 高德相关请求由 main agent 调用 `amap_worker` 处理。
- main agent 调用 worker 时可以生成 Worker Task Package。
- Worker Task Package 可以传递任务、用户原始消息、自然语言交接信息、轻量摘要和最近几轮对话拼接位置。
- worker 可以自行完成领域参数抽取和缺参判断。
- `amap_worker` 内部可以调用 `api / chain / code` Skill。
- worker 执行结果可以回到 main agent。
- main agent 最终生成面向用户的自然回答。
- 缺参、多轮补充、失败和 AG-UI 中间步骤都可运行。

## 一、预置 Worker

第三阶段预置一个真实 worker：

| worker_id | 用途 |
| --- | --- |
| `amap_worker` | 高德 Web 服务领域 worker，处理地址、路线、天气、行政区域、IP 定位、坐标转换 |

### 1.1 amap_worker Skill

| skill_id | 类型 | 用途 |
| --- | --- | --- |
| `amap_geocode` | `api` | 地址转经纬度、adcode、citycode |
| `amap_regeocode` | `api` | 经纬度转结构化地址 |
| `amap_weather` | `api` | 根据 adcode 查询实时天气或天气预报 |
| `amap_district_search` | `api` | 行政区域查询 |
| `amap_ip_location` | `api` | IPv4 地址定位 |
| `amap_coordinate_convert` | `api` | 坐标系转换 |
| `amap_route_driving` | `api` | 驾车路径规划 |
| `amap_route_walking` | `api` | 步行路径规划 |
| `amap_route_transit` | `api` | 公交路径规划 |
| `amap_route_distance` | `api` | 距离测量 |
| `travel_briefing_formatter` | `code` | 路线和天气结果格式化 |
| `amap_route_weather_plan` | `chain` | 地址解析 → 路线 → 目的地天气 → 出行建议 |

## 二、测试原则

第三阶段测试分四层：

1. Worker 资产加载测试。
2. Worker Skill 迁移测试。
3. main → worker → skill 全链路测试。
4. AG-UI 中间步骤测试。

测试必须保持：

- 单轮简单问题使用新 `session_id`。
- 多轮测试每组使用独立新 `session_id`，组内复用同一个 `session_id`。
- 高德领域请求只要可以识别 worker，就应进入 `amap_worker`。
- main agent 不直接调用高德 Skill。
- 用户消息使用正常表达。
- 用户侧输出不能暴露 worker 内部 Prompt、Adapter、schema、API Key、异常堆栈。
- main 直接 `ask_user → awaiting_user` 只用于无法判断应交给哪个 worker 的场景。
- 用户补充信息后，必须能回到 `call_agent(amap_worker)` 链路。

## 三、统一请求格式

所有 Agent 链路测试通过 `/agent/run` 或 `/agent/run/stream` 触发。

```json
{
  "user_id": "worker_test_user",
  "agent_id": "main",
  "session_id": null,
  "request_id": "case_worker_001",
  "message": "我从杭州市西湖区文三路开车去杭州市上城区湖滨银泰，帮我看路线和目的地天气。",
  "metadata": {}
}
```

## 四、轨迹采集

每个用例需要采集：

```text
request
main_agent_run
main_agent_run_step[]
worker_agent_run
worker_agent_run_step[]
main_tool_call[]
worker_tool_call[]
final_response
session_state
ag_ui_events[]
```

检查重点：

```text
main tool_call.tool_name = amap_worker
worker tool_call.tool_name = 具体 skill_id
main agent_run_step.action_detail 包含 Worker Task Package
main final_state = completed 或 awaiting_user
worker final_state = completed 或 awaiting_user
```

AG-UI 流式用例至少检查：

```text
RUN_STARTED
STATE_SNAPSHOT
STEP_STARTED
STATE_DELTA
TOOL_CALL_START
TOOL_CALL_ARGS
TOOL_CALL_END
TEXT_MESSAGE_START
TEXT_MESSAGE_CONTENT
TEXT_MESSAGE_END
RUN_FINISHED
```

## 五、资产和迁移测试

### 5.1 Worker 资产加载

检查点：

- `agents/workers/amap_worker/` 存在。
- 必要文件齐全：

```text
SOUL.md
Agent.md
Instruction.md
tools.md
output.md
harness.md
```

- `amap_worker` 可以构建 AgentDefinition。
- `amap_worker.skill_registry` 可以列出自己的 SkillCatalog。

### 5.2 main agent 不直接持有高德 Skill

检查点：

- `agents/main/skills/` 只保留通用 Skill。
- `content_extract` 仍可由 main agent 使用。
- 以下 Skill 不应出现在 main SkillCatalog：

```text
amap_geocode
amap_regeocode
amap_weather
amap_route_driving
amap_route_weather_plan
```

### 5.3 amap_worker 持有高德 Skill

检查点：

- `amap_worker` SkillCatalog 包含第一批高德 Skill。
- 每个 Skill 都有 `SKILL.md` 和 `schema.json`。
- 每个 `schema.json` 的 `skill_id` 与目录名一致。
- 每个 Skill 的输入和输出 schema 可通过校验。

### 5.4 Worker Task Package 结构

用户输入：

```text
我从杭州市西湖区文三路开车去杭州市上城区湖滨银泰，帮我看路线和目的地天气。
```

检查点：

- main 的 `call_agent` action_detail 包含 `worker_id = amap_worker`。
- action_detail 包含 `task`。
- action_detail 包含 `handoff_context`，内容是 main 已理解到的自然语言关键信息。
- action_detail 包含 `context_package.user_message`。
- action_detail 包含 `context_package.conversation_summary`。
- action_detail 包含 `context_package.recent_turns`。
- action_detail 包含 `handoff.reason`。
- action_detail 包含 `handoff.main_known_missing_params`。
- action_detail 包含 `context_package.retrieved_context`，第三阶段允许为空数组。
- action_detail 包含 `context_package.long_term_memory`，第三阶段允许为空数组。
- action_detail 包含 `context_package.memory_palace_refs`，第三阶段允许为空数组。
- action_detail 包含 `output_contract.final_answer_owner = main_agent`。

第三阶段不要求：

- LLM 压缩摘要。
- conversation_record 向量召回。
- 长期记忆召回。
- 记忆宫殿召回。

### 5.5 Worker 缺参回流

用户输入：

```text
帮我查一下去杭州市上城区湖滨银泰的路线和天气。
```

检查点：

- main 识别出高德领域后，应调用 `amap_worker` 获取领域缺参判断。
- 如果 main 调用 `amap_worker`，worker 可以返回 `missing_params`。
- main Harness 可以识别 worker 缺参结果。
- 下一轮 Loop 先判断 main 是否能从已有上下文补齐。
- 如果不能补齐，main 追问用户。
- 用户补充后，main 重新调用 `amap_worker`，并带上 `previous_worker_result` 或相关缺参上下文。

## 六、单轮 Worker 测试

### 6.1 地址解析

用户输入：

```text
帮我解析一下地址：杭州市西湖区文三路，城市是杭州。
```

期望链路：

```text
main.new_request
  → call_agent(amap_worker)
  → worker.call_skill(amap_geocode)
  → worker.answer_user
  → main.answer_user
  → completed
```

检查点：

- main `tool_call.tool_name = amap_worker`。
- worker `tool_call.tool_name = amap_geocode`。
- worker 输出包含 `location` 和 `adcode`。
- 最终回答自然说明解析结果。

### 6.2 实时天气

用户输入：

```text
查询 adcode 330106 的实时天气。
```

期望链路：

```text
main.call_agent(amap_worker)
  → worker.call_skill(amap_weather)
  → completed
```

检查点：

- main 不直接调用 `amap_weather`。
- worker 调用 `amap_weather`。
- 输出包含天气、温度、风向、湿度。

### 6.3 驾车路径规划

用户输入：

```text
帮我查一下从 120.123456,30.123456 到 120.654321,30.654321 的驾车路线。
```

期望链路：

```text
main.call_agent(amap_worker)
  → worker.call_skill(amap_route_driving)
  → completed
```

检查点：

- worker 使用 `amap_route_driving`。
- 输出包含距离、预计时间和路线摘要。

### 6.4 地址到路线天气建议

用户输入：

```text
我从杭州市西湖区文三路开车去杭州市上城区湖滨银泰，帮我看路线和目的地天气。
```

期望链路：

```text
main.call_agent(amap_worker)
  → worker.call_skill(amap_route_weather_plan)
  → completed
```

检查点：

- worker 使用 chain skill `amap_route_weather_plan`。
- chain 内部包含地址解析、路线、天气和格式化结果。
- worker 返回结构化领域结果。
- main 最终回答面向用户，不暴露内部链路名。

## 七、缺参与多轮测试

### 7.1 缺少出发地

用户输入：

```text
帮我查一下去杭州市上城区湖滨银泰的路线和天气。
```

期望链路：

```text
main.call_agent(amap_worker)
  → worker.missing_params
  → main.harness
  → main.ask_user
  → awaiting_user
```

检查点：

- 追问聚焦缺少出发地。
- 不使用当前位置、默认地点或模型猜测地点补齐参数。
- worker 返回 `missing_params`。
- main 根据 worker 缺参结果追问用户。

### 7.2 多轮补充完成

轮次 1：

```text
帮我规划一下明天开车出行，顺便看天气。
```

期望链路：

```text
main.call_agent(amap_worker) → worker.missing_params → main.ask_user → awaiting_user
```

轮次 2：

```text
我从杭州市西湖区文三路去杭州市上城区湖滨银泰。
```

期望链路：

```text
awaiting_user
  → main.call_agent(amap_worker)
  → worker.call_skill(amap_route_weather_plan)
  → completed
```

检查点：

- 第二轮复用同一个 `session_id`。
- Runtime 可以承接上一轮待补充意图。
- 第二轮进入 `amap_worker`。
- worker 进入 `amap_route_weather_plan`。

### 7.3 地址无法解析

用户输入：

```text
我从一个不存在的火星基地开车去杭州市上城区湖滨银泰，帮我看路线和天气。
```

允许链路：

```text
main.call_agent(amap_worker) → worker.missing_params → main.ask_user → awaiting_user
```

或：

```text
main.call_agent(amap_worker) → worker.call_skill(amap_route_weather_plan) → failed
```

检查点：

- 用户侧自然说明地址无法确认。
- 不暴露高德原始错误堆栈。
- 不编造路线结果。

## 八、AG-UI 流式测试

### 8.1 Worker 调用步骤

请求：

```text
查询 adcode 330106 的实时天气。
```

接口：

```text
POST /agent/run/stream
```

检查点：

- 输出 `TOOL_CALL_START`，`toolName = amap_worker`。
- 输出 worker 内部 Skill 步骤，至少包含 `amap_weather`。
- 输出 `TOOL_CALL_END`，显示已拿到 worker 结果。
- 最终回答通过 `TEXT_MESSAGE_CONTENT` 流式输出。

### 8.2 Chain Skill 步骤

请求：

```text
我从杭州市西湖区文三路开车去杭州市上城区湖滨银泰，帮我看路线和目的地天气。
```

检查点：

- 展示 `amap_worker` 调用。
- 展示 `amap_route_weather_plan` 或其内部摘要。
- 展示“已拿到结果”类步骤。
- 不展示 API Key。
- 不展示完整原始接口响应。

## 十、验收要求

第三阶段测试通过条件：

- `amap_worker` 可以被 main agent 扫描和调用。
- 高德 Skill 不再出现在 main SkillCatalog。
- 高德 Skill 出现在 `amap_worker` SkillCatalog。
- main `call_agent` 真实调用 worker，不再走 mock worker executor。
- main `call_agent` 可以生成 Worker Task Package。
- Worker Task Package 中第三阶段可用上下文正常传递。
- Worker Task Package 中第四阶段检索字段位置存在。
- worker 内部可以调用真实 Skill。
- worker Skill 结果进入 worker Harness。
- worker 返回结果进入 main Harness。
- 缺参时可以正确追问。
- 多轮补充后可以进入正确 worker 链路。
- AG-UI 可以展示 worker 和 Skill 中间步骤。
- 用户侧输出自然，不暴露内部实现细节。
