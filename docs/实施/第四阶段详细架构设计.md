# 第四阶段详细架构设计

## 概述

本文档描述第四阶段「会话上下文压缩闭环」的实施架构。

第一阶段跑通主 Agent Loop。

第二阶段跑通真实 Skill。

第三阶段跑通真实 worker，并引入事件级上下文：

- main / worker 共享同一个 `event_id`。
- 事件结束后可以沉淀 `agent_event_context.summary`。
- Loop 前可以通过 `EventContextPreloader` 注入命中事件内的一轮或多轮原始对话。

第四阶段不建设完整个人记忆系统。

第四阶段的目标是解决业务 Agent 在当前会话中上下文变长后的连续性问题：

```text
当前会话原始对话越来越长
  ↓
Prompt 预算接近上限
  ↓
压缩较早对话为 conversation_summary
  ↓
保留最近对话原文 recent_turns
  ↓
必要时通过 event_id 找回被压缩部分中的相关原始对话 event_dialogues
  ↓
Context Engineer 去重后拼装 Prompt
```

## 一、第四阶段范围

### 要实现

第四阶段需要实现：

- 当前会话对话记录可持久化。
- 每个 `conversation_turn` 拥有稳定 `turn_id` 或可等价追踪的 run 标识。
- 基于 `user_id + agent_id + session_id` 维护会话上下文。
- 估算当前会话原始对话窗口的 Prompt token 占用。
- 当原始对话窗口接近配置阈值时，触发会话摘要压缩。
- 将较早对话压缩进 `session_state.conversation_summary`。
- 保留最近若干轮原始对话作为 `recent_turns`。
- `EventContextPreloader` 继续提供 `event_dialogues`。
- Context Engineer 在 `recent_turns` 和 `event_dialogues` 之间按 `conversation_turn` 精确去重。
- Prompt 拼装时形成清晰的上下文层级。

### 不实现

第四阶段暂不实现：

- 记忆宫殿 Provider。
- 记忆宫殿召回。
- 记忆宫殿管理接口。
- `memory_manager` worker。
- Hook 触发后台 `memory_manager`。
- 跨会话长期记忆召回。
- 跨用户召回。
- 全局事件检索。
- 用户画像或稳定偏好自动沉淀。
- 向量检索。

记忆宫殿更适合个人助手型 Agent 框架，不进入当前业务型 Agent 框架的第四阶段。

相关内容记录到 `docs/技术方案/技术债务.md`。

## 二、核心概念

第四阶段上下文分四层：

```text
conversation_summary
recent_turns
event_dialogues
current_user_message
```

### 2.0 conversation_turn

`conversation_turn` 是一轮完整用户可见对话。

它不是 Loop step，也不是模型内部多轮消息。

一条 `conversation_turn` 对应：

```text
用户一次输入
  +
Agent 对这次输入最终给用户的一次回答或追问
```

它和以下字段相关：

```text
user_id
agent_id
session_id
event_id
run_id
turn_id
user_message
assistant_message
final_state
created_at
token_estimate
summary_batch_id
summarized_at
```

作用域：

```text
user_id + agent_id + session_id
```

说明：

- main 有 main 的 `conversation_turns`。
- worker 有 worker 的 `conversation_turns`。
- 同一个 session 下，不同 agent 的 `conversation_turns` 分开管理。
- 同一个 event 下，main 和 worker 可以各自有 `conversation_turns`。
- `event_id` 只表示这轮属于哪件事，不是 `conversation_turns` 的主作用域。

示例：

```text
conversation_turn_001
user: 我从文三路去湖滨银泰，帮我看路线和天气
assistant: 已为你查到路线和天气...

conversation_turn_002
user: 那目的地现在天气怎么样？
assistant: 目的地当前天气是...
```

Loop step 和 `conversation_turn` 的关系：

```text
conversation_turn_001
  ├── main loop step 1: call_agent
  ├── worker loop step 1: call_skill
  ├── worker loop step 2: answer_user
  └── main loop step 2: answer_user
```

用户只看到一轮问答，但 Runtime 内部可能执行多个 Loop step。

### 2.1 conversation_summary

`conversation_summary` 是当前 session 中较早对话的压缩结果。

作用域：

```text
user_id + agent_id + session_id
```

存储位置：

```text
session_state.conversation_summary
```

内容保留：

- 当前会话主要目标。
- 已确认参数。
- 已完成动作。
- 待补充问题。
- 当前结论。
- 重要约束。

边界：

- 只服务当前 session。
- 不作为长期记忆。
- 不跨 session 复用。
- 不写入记忆宫殿。
- 不用于用户画像。

### 2.2 recent_turns

`recent_turns` 是当前 session 中最近若干个 `conversation_turns` 的原始对话。

它不压缩，直接作为原文进入 Prompt。

作用：

- 保留最近上下文的完整表达。
- 避免每轮都依赖摘要。
- 给模型看到最近的问答细节。

来源：

```text
conversation_record
或 agent_run 中可还原的一轮 user / assistant 对话
```

推荐结构：

```json
{
  "turn_id": "turn_xxx",
  "run_id": "run_xxx",
  "event_id": "evt_xxx",
  "agent_id": "main",
  "user": "用户输入",
  "assistant": "Agent 回答",
  "token_estimate": 320
}
```

进入 Prompt 时不暴露 `turn_id / run_id / event_id`。

这些 ID 只用于 Runtime 去重和追踪。

### 2.3 event_dialogues

`event_dialogues` 来自第三阶段的 `EventContextPreloader`。

它表示当前请求命中的事件内一轮或多轮原始对话。

作用：

- 当某个事件的细节已经不在 `recent_turns` 中，甚至已经被压缩进 `conversation_summary` 中时，把相关原始对话补回来。
- 支持用户对上一个事件或当前会话中某个历史事件继续追问。

边界：

- 只在当前会话内做事件相关性判断。
- 只注入命中事件内的一轮或多轮对话。
- 不注入 `event_id`、事件标题、summary、worker / Skill 执行摘要。
- 与 `recent_turns` 重叠时必须去重。

### 2.4 current_user_message

当前用户输入永远保留。

它不参与压缩，不被去重，不被摘要替代。

## 三、为什么不每轮总结

第四阶段不采用“每轮完成后都总结”的策略。

原因：

- 每轮总结会增加模型调用成本。
- 每轮总结会带来摘要漂移。
- 当前会话很短时，原始对话比摘要更可靠。
- 第三阶段已经有事件内对话召回，过早摘要会让上下文层级混乱。

第四阶段采用“空间不足再压缩”的策略：

```text
原始对话窗口 token 未接近阈值
  → 不总结
  → 保留原始 recent_turns

原始对话窗口 token 接近阈值
  → 压缩较早对话
  → 更新 conversation_summary
  → 保留最新若干轮原始对话
```

## 四、Prompt 预算策略

第四阶段需要引入会话上下文预算。

建议配置：

```text
SESSION_CONTEXT_MAX_TOKENS=6000
SESSION_SUMMARY_TRIGGER_RATIO=0.8
SESSION_SUMMARY_TARGET_AFTER_COMPRESSION_RATIO=0.35
SESSION_RECENT_TURN_MIN_COUNT=4
SESSION_RECENT_TURN_MAX_COUNT=8
SESSION_SUMMARY_TARGET_TOKENS=800
SESSION_TURN_COMPRESS_BATCH_SIZE=6
```

含义：

| 配置 | 说明 |
| --- | --- |
| `SESSION_CONTEXT_MAX_TOKENS` | 当前会话上下文可占用的最大 token 预算 |
| `SESSION_SUMMARY_TRIGGER_RATIO` | 达到多少比例后触发压缩 |
| `SESSION_SUMMARY_TARGET_AFTER_COMPRESSION_RATIO` | 压缩后未压缩原始对话窗口目标比例 |
| `SESSION_RECENT_TURN_MIN_COUNT` | 无论是否压缩，都至少保留最近几轮原始对话 |
| `SESSION_RECENT_TURN_MAX_COUNT` | 常规情况下最多保留的最近原始对话轮数 |
| `SESSION_SUMMARY_TARGET_TOKENS` | 会话摘要目标长度 |
| `SESSION_TURN_COMPRESS_BATCH_SIZE` | 每次最多压缩的较早 conversation_turn 数量 |

初始建议：

```text
历史上下文窗口：6000 tokens
触发水位：6000 * 0.8 = 4800 tokens
压缩后目标水位：6000 * 0.35 = 2100 tokens
```

压缩后的目标水位允许落在 30% 到 40% 区间。

第一版默认使用 35%。

这样可以避免刚压缩完，下一两轮又立刻触发二次压缩。

第四阶段可以先使用近似 token 估算。

推荐第一版估算：

```text
中文字符数 / 1.5 + 英文单词数
```

后续可替换为模型 tokenizer。

## 五、上下文拼装顺序

Context Engineer 拼装 Prompt 时使用以下顺序：

```text
Agent files
Skill / Worker catalog
conversation_summary
recent_turns
event_dialogues_deduped
current_user_message
previous_action_result
action_history
output_schema
```

核心规则：

1. 当前用户输入永远保留。
2. 最近几轮原始对话优先保留。
3. 事件召回只补充不在最近几轮里的相关原始对话。
4. 会话摘要只覆盖更早的、已经被压缩的部分。
5. 如果某个 turn 已经作为 raw dialogue 注入，不再从 event_dialogues 注入一次。
6. `turn_id / run_id / event_id` 只用于 Runtime 去重，不进入 Prompt。

## 六、去重策略

第四阶段必须解决 `recent_turns` 和 `event_dialogues` 的重叠问题。

去重必须是纯代码精确去重，目标是毫秒级完成。

不做：

- 语义相似度。
- 模型判断。
- 模糊匹配。
- 和 `conversation_summary` 去重。

只需要避免 raw dialogue 和 raw dialogue 重复。

### 6.1 去重键

第一版去重键优先级：

```text
1. 有 turn_id：使用 turn_id
2. 没有 turn_id：使用 run_id
3. 都没有：使用 sha1(agent_id + user + assistant)
```

说明：

- `turn_id` 和 `run_id` 都是 Runtime 内部字段，不进入 Prompt。
- 内容指纹只做完全一致去重，不做相似度判断。
- `conversation_summary` 不参与去重，因为它是压缩层，和原始细节层允许语义重叠。

### 6.2 去重规则

```text
recent_turns
  ↓
recent_keys = {dialogue_key(turn) for turn in recent_turns}
  ↓
遍历 event_dialogues
  ↓
如果 dialogue_key(event_dialogue) in recent_keys：跳过
  ↓
否则加入 event_dialogues_deduped
```

复杂度：

```text
O(len(recent_turns) + len(event_dialogues))
```

第一版数据量很小，通常：

```text
recent_turns: 4 到 8 轮
event_dialogues: 1 到 3 轮
```

因此应保持毫秒级。

### 6.3 重叠示例

场景：

- 用户刚刚完成路线查询。
- 下一轮用户问：“那附近停车方便吗？”
- 上一个事件对话仍在 `recent_turns` 中。
- `EventContextPreloader` 也命中了上一个事件。

处理：

```text
recent_turns 已包含上一轮路线问答
event_dialogues 命中同一轮路线问答
  ↓
event_dialogues_deduped 跳过该轮
```

### 6.4 被压缩后的补细节示例

场景：

- 用户会话很长。
- 早期路线查询已经被压缩进 `conversation_summary`。
- `recent_turns` 不再包含路线查询原文。
- 用户追问：“那次路线的目的地天气现在怎么样？”

处理：

```text
conversation_summary 提供早期任务概貌
EventContextPreloader 命中早期事件
event_dialogues 注入早期路线查询原文
  ↓
模型拿到摘要 + 原始细节
```

## 七、会话摘要压缩流程

第四阶段新增轻量 Hook：

```text
SessionContextCompressionRequested
```

触发时机：

```text
每轮 Run 完成
  ↓
记录本轮 conversation_turn
  ↓
Hook 检查当前 session 未压缩原始 conversation_turns token
  ↓
未达到阈值：不压缩
  ↓
达到阈值：压缩较早 conversation_turns
```

注意：

每轮完成后可以检查预算，但不是每轮都调用模型总结。

### 7.1 压缩输入

压缩输入包括：

- 现有 `session_state.conversation_summary`。
- 待压缩的较早 `conversation_turns`。
- 当前 session 已确认的 pending / missing 状态。

不包括：

- 当前最新保留的 recent turns。
- 其他 session 的对话。
- 其他用户的对话。
- 记忆宫殿内容。

### 7.2 压缩输出

摘要输出写回：

```text
session_state.conversation_summary
```

摘要内容要求：

- 保留用户当前目标。
- 保留已确认参数。
- 保留已完成动作。
- 保留待补充问题。
- 保留当前结论。
- 保留仍可能影响后续回答的约束。

摘要不应包含：

- 全量工具原始响应。
- API Key。
- 内部 Prompt。
- 无关闲聊细节。
- 已被明确废弃的旧目标。

### 7.3 压缩状态标记

如果引入 `conversation_record` 表，建议增加：

```text
turn_id
run_id
event_id
agent_id
role
content
token_estimate
summary_batch_id
summarized_at
created_at
```

如果第四阶段先复用 `agent_run`，则需要在 repository 层维护等价的 turn 选择和压缩边界。

### 7.4 压缩选择规则

压缩时优先保护最近原始对话。

规则：

```text
1. 计算当前未压缩 raw conversation_turns token。
2. 如果 raw_tokens < SESSION_CONTEXT_MAX_TOKENS * SESSION_SUMMARY_TRIGGER_RATIO：
   不压缩。
3. 如果 raw_tokens >= SESSION_CONTEXT_MAX_TOKENS * SESSION_SUMMARY_TRIGGER_RATIO：
   从最早的未压缩 conversation_turn 开始选择待压缩 turns。
4. 压缩后目标是让剩余 raw turns token 尽量回到：
   SESSION_CONTEXT_MAX_TOKENS * SESSION_SUMMARY_TARGET_AFTER_COMPRESSION_RATIO。
5. 无论如何，最近 SESSION_RECENT_TURN_MIN_COUNT 个 conversation_turns 必须保留原文。
6. 如果最近 4 轮本身已经超过 35% 水位，也不压缩这 4 轮。
```

以默认配置为例：

```text
SESSION_CONTEXT_MAX_TOKENS=6000
SESSION_SUMMARY_TRIGGER_RATIO=0.8
SESSION_SUMMARY_TARGET_AFTER_COMPRESSION_RATIO=0.35

触发压缩水位：4800 tokens
压缩后目标水位：2100 tokens
```

这意味着：

- 原始对话窗口到 4800 tokens 左右才压缩。
- 压缩较早部分。
- 保留最近 4 到 8 个 `conversation_turns`。
- 目标是让未压缩 raw turns 回到约 2100 tokens。
- 如果最近 4 轮已经超过 2100 tokens，仍然保护最近 4 轮，不强行压缩。

## 八、Runtime 改造

### 8.1 SessionContextManager

新增：

```text
runtime/context/session_context_manager.py
```

职责：

- 读取当前 session 的历史 `conversation_turns`。
- 估算 recent turns token。
- 判断是否需要触发压缩。
- 选择待压缩 `conversation_turns`。
- 生成 recent_turns。
- 调用去重器合并 event_dialogues。

### 8.2 SessionContextCompressor

新增：

```text
runtime/context/session_context_compressor.py
```

职责：

- 接收旧摘要和待压缩 `conversation_turns`。
- 调用模型生成新摘要。
- 控制摘要目标 token。
- 写回 `session_state.conversation_summary`。

### 8.3 ContextDeduplicator

新增：

```text
runtime/context/context_deduplicator.py
```

职责：

- 对 `recent_turns` 和 `event_dialogues` 去重。
- 输出 `event_dialogues_deduped`。
- 确保 Prompt 不重复注入同一轮原始对话。

### 8.4 RuntimeHook

Hook 新增事件：

```text
SessionContextCompressionRequested
```

Hook 边界：

- 不参与 Loop 决策。
- 不影响当前用户回答。
- 预算未到阈值时不调用模型。
- 压缩失败只记录日志，不回滚 Agent Run。

## 九、main 与 worker 的处理

会话摘要作用域包含 `agent_id`。

```text
user_id + agent_id + session_id
```

因此：

- main 拥有自己的 `conversation_summary`。
- worker 拥有自己的 `conversation_summary`。
- main 与 worker 不共享同一条会话摘要。
- main 调 worker 时，仍通过 Worker Task Package 传递必要的父上下文。
- worker 可以读取自己的 recent turns 和事件内对话。

第四阶段不要求 worker 做跨事件检索。

worker 仍使用 main 当前 `event_id`。

---

## 十、与第三阶段事件上下文的关系

第三阶段已经实现：

```text
EventContextPreloader
  ↓
event_dialogues
```

第四阶段在此基础上增加：

```text
recent_turns
  ↓
ContextDeduplicator
  ↓
event_dialogues_deduped
```

关系：

| 上下文 | 作用 |
| --- | --- |
| `conversation_summary` | 当前 session 较早对话压缩层 |
| `recent_turns` | 当前 session 最近原始对话层 |
| `event_dialogues` | 命中事件的原始细节召回层 |
| `event_dialogues_deduped` | 去重后真正进入 Prompt 的事件对话 |

重点：

`event_dialogues` 的价值不是替代 recent turns。

它的价值是当细节被压缩进 summary 后，按事件把相关原文补回来。

## 十一、验收标准

第四阶段通过条件：

- 短会话不触发摘要模型调用。
- 未压缩 raw `conversation_turns` 达到约 4800 tokens 时触发摘要压缩。
- 压缩后 `session_state.conversation_summary` 被更新。
- 压缩后未压缩 raw turns 尽量回到约 2100 tokens。
- 压缩后仍保留最近 4 到 8 轮原始对话。
- 最近 4 轮原始对话必须保护，即使超过 2100 tokens。
- Prompt 中包含 `conversation_summary`。
- Prompt 中包含 `recent_turns`。
- Prompt 中可包含去重后的 `event_dialogues_deduped`。
- `recent_turns` 与 `event_dialogues` 重叠时通过纯代码精确去重，不重复注入。
- 事件对话已经被压缩出 recent turns 时，可以通过事件召回补回原文。
- 当前用户输入永远保留。
- 摘要只服务当前 session，不跨 session 复用。
- 不接入记忆宫殿。
- 不接入 `memory_manager`。
- 不做跨用户或跨会话召回。
