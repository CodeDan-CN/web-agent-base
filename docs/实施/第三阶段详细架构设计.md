# 第三阶段详细架构设计

## 概述

本文档描述第三阶段「Worker Agent 闭环」的实施架构。

第一阶段已经跑通主 Agent 的 Loop 骨架。

第二阶段已经跑通主 Agent 调用真实 Skill 的闭环。

第三阶段的目标是把 `call_agent` 从 mock worker executor 替换为真实 worker agent 运行机制。

第三阶段完成后，main agent 不再直接持有高德领域 Skill，而是根据用户意图调用 `amap_worker`。`amap_worker` 拥有自己的 Agent 资产、SkillCatalog、Harness 和输出规范，并在 worker 内部调用高德基础 API Skill，最后把结构化结果返回 main agent，由 main agent 面向用户生成最终回答。

## 一、第三阶段范围

### 要实现

第三阶段需要实现：

- `agents/workers/` 目录扫描。
- Worker AgentDefinition 加载。
- Worker Agent 拥有独立 Markdown 资产。
- Worker Agent 拥有独立 `skills/` 目录和 SkillRegistry。
- main agent 的 `call_agent` 使用真实 `worker_id` 和 Worker Task Package 调用 worker。
- Worker Task Package 可以传递当前请求、任务描述、自然语言交接信息、轻量会话摘要和最近几轮对话。
- 为第四阶段的检索上下文、长期记忆和记忆宫殿结果预留字段。
- worker 内部复用第二阶段 SkillExecutor。
- worker 内部执行自己的 LoopDecider、ActionExecutor、HarnessEvaluator 和 StateManager。
- worker 结果以结构化 `ActionResult` 返回 main agent。
- main agent Harness 评估 worker 返回结果。
- `tool_call` 记录真实 worker 调用。
- AG-UI 输出真实 worker 调用步骤。
- 高德相关 Skill 从 `agents/main/skills/` 下沉到 `agents/workers/amap_worker/skills/`。

### 不实现

第三阶段暂不实现：

- worker 自主创建 Skill。
- worker 自主创建其他 worker。
- worker 间任意互相调用。
- 多 worker 并行调度。
- Worker 权限系统。
- Worker 资源隔离。
- Worker 独立部署。
- Worker 独立数据库。
- Agent 市场或 Skill 市场。
- 记忆闭环。
- 记忆宫殿管理接口。
- LLM 对话压缩摘要。
- conversation_record 向量召回。
- 长期记忆召回。
- 记忆宫殿召回。

第三阶段只解决 main agent 调用单个真实 worker agent 的闭环。

第三阶段只保证 worker 上下文传递协议和拼接位置存在。

完整压缩、检索和记忆上下文传递进入第四阶段。

## 二、目标链路

第三阶段需要验证以下链路：

```text
new_request
  → call_agent(amap_worker)
  → worker.new_request
  → worker.call_skill
  → worker.ready_to_plan
  → worker.answer_user
  → main.ready_to_plan
  → main.answer_user
  → completed
```

缺参链路：

```text
new_request
  → call_agent(amap_worker)
  → worker.missing_params
  → main.harness
  → main.loop
  → main.ask_user
  → awaiting_user
```

多轮补充链路：

```text
new_request
  → call_agent(amap_worker)
  → worker.missing_params
  → main.ask_user
  → awaiting_user

awaiting_user
  → call_agent(amap_worker)
  → worker.call_skill
  → worker.ready_to_plan
  → worker.answer_user
  → main.ready_to_plan
  → main.answer_user
  → completed
```

说明：

- main agent 只决定是否调用 worker，不直接理解高德 API 细节。
- 只要 main agent 已经能判断请求属于某个 worker 的领域，第一轮应优先进入该 worker，由 worker 判断领域缺参。
- worker agent 负责在自己的领域内选择 Skill。
- worker agent 负责判断领域任务是否缺少必要信息。
- worker 的 `answer_user` 在第三阶段不直接面向最终用户，而是生成给 main agent 的结构化领域结果。
- 最终用户回答仍由 main agent 输出。

## 四、主 Agent 职责调整

第三阶段开始，main agent 的职责收窄为调度和最终回答。

main agent 保留：

```text
agents/main/skills/content_extract/
```

main agent 移除或不再加载：

```text
agents/main/skills/amap_geocode/
agents/main/skills/amap_weather/
agents/main/skills/amap_direction_driving/
agents/main/skills/travel_briefing_formatter/
agents/main/skills/amap_route_weather_plan/
```

这些高德相关能力迁移到：

```text
agents/workers/amap_worker/skills/
```

main agent 只需要知道：

- `amap_worker` 可以处理地址解析、天气、路径规划、行政区域、IP 定位、坐标转换等高德能力。
- 如果用户请求属于高德领域，选择 `call_agent`，目标 worker 为 `amap_worker`。
- main agent 不直接拼接高德 API 参数。
- main agent 不直接调用高德 Skill。

## 五、amap_worker 设计

### 5.1 Worker 定位

`amap_worker` 是高德 Web 服务领域 worker。

职责：

- 理解用户的地图、地址、路线、天气、行政区划、IP 定位、坐标转换诉求。
- 判断是否需要追问。
- 在自己的 SkillCatalog 中选择合适 Skill。
- 将 Skill 执行结果组织成结构化领域结果。
- 不直接输出最终用户回答的完整文案。

### 5.2 Worker 资产

`amap_worker` 需要包含：

```text
agents/workers/amap_worker/
├── SOUL.md
├── Agent.md
├── Instruction.md
├── tools.md
├── output.md
├── harness.md
├── prompts/
│   ├── fragments.md
│   └── examples.md
└── skills/
```

各文件职责与 main agent 保持一致。

内容差异：

- `Agent.md` 描述高德领域职责和边界。
- `Instruction.md` 描述高德任务处理 SOP。
- `tools.md` 描述 worker 内部 Skill 使用规则。
- `output.md` 要求 worker 输出结构化领域结果，不面向最终用户发挥。
- `harness.md` 评估高德 Skill 结果是否足以返回 main agent。

## 六、amap_worker 第一批 Skill

第三阶段第一批只使用高德 Web 服务基础 API 和已确认可用的天气查询 API。

| skill_id | 类型 | 用途 |
| --- | --- | --- |
| `amap_geocode` | `api` | 地址转经纬度、adcode、citycode |
| `amap_regeocode` | `api` | 经纬度转结构化地址 |
| `amap_weather` | `api` | 根据 adcode 查询实时天气或天气预报 |
| `amap_district_search` | `api` | 行政区域查询 |
| `amap_ip_location` | `api` | IPv4 地址定位 |
| `amap_coordinate_convert` | `api` | 坐标系转换 |
| `amap_route_driving` | `api` | 驾车路径规划 |
| `amap_route_walking` | `api` | 步行路径规划 |
| `amap_route_transit` | `api` | 公交路径规划 |
| `amap_route_distance` | `api` | 距离测量 |
| `travel_briefing_formatter` | `code` | 路线和天气结果格式化 |
| `amap_route_weather_plan` | `chain` | 地址解析 → 路线 → 目的地天气 → 出行建议 |

说明：

- `amap_direction_driving` 在第三阶段建议改名为 `amap_route_driving`。
- `amap_route_weather_plan` 是 worker 内部 chain skill，不属于 main agent。
- API Key 仍然只来自 `.env`。
- 不把 API Key 写入 Prompt、Skill 资产或 AG-UI 事件。

## 七、Worker Task Package 与上下文边界

Worker Task Package 是 main agent 派发给 worker 的轻量任务包。

它解决的问题不是 worker 如何执行 Skill，而是 main agent 传给 worker 什么、不给什么、边界在哪里。

main agent 不负责一次性给齐 worker 需要的所有领域参数。

worker 必须基于任务包自行完成领域参数抽取、缺参判断、Skill 选择和结构化结果组织。

第三阶段必须先把这个协议跑通。

第四阶段再补齐压缩摘要、检索上下文、长期记忆和记忆宫殿结果。

### 7.1 LoopDecider 输出结构

main agent 调用 worker 时，LoopDecider 输出：

```json
{
  "action": "call_agent",
  "action_detail": {
    "worker_id": "amap_worker",
    "task": "用户想查询一次路线和目的地天气",
    "handoff_context": "用户提到从杭州市西湖区文三路去杭州市上城区湖滨银泰，倾向于开车，并希望查看目的地天气。",
    "context_package": {
      "user_message": "用户原始请求",
      "conversation_summary": "当前 session 的轻量滚动摘要",
      "recent_turns": [],
      "retrieved_context": [],
      "long_term_memory": [],
      "memory_palace_refs": [],
      "previous_worker_result": null
    },
    "handoff": {
      "reason": "用户请求属于高德地图领域",
      "main_known_missing_params": []
    },
    "output_contract": {
      "format": "structured_result",
      "final_answer_owner": "main_agent"
    }
  },
  "reason": "需要高德领域能力处理"
}
```

### 7.2 第三阶段实际传递内容

第三阶段实际传递：

| 字段 | 是否实现 | 说明 |
| --- | --- | --- |
| `worker_id` | 是 | 目标 worker |
| `task` | 是 | main agent 对任务的简短描述，不要求包含全部领域参数 |
| `handoff_context` | 是 | main agent 已理解到的关键信息，使用自然语言描述，可为空 |
| `context_package.user_message` | 是 | 当前用户原始消息 |
| `context_package.conversation_summary` | 是 | 当前 `session_state.conversation_summary`，只是轻量滚动摘要 |
| `context_package.recent_turns` | 是 | 最近几轮对话的拼接位置，第三阶段可以先从现有 run/session 信息构造简版 |
| `context_package.previous_worker_result` | 是 | 上一次相关 worker 结果，可为空 |
| `handoff.reason` | 是 | main agent 为什么把任务交给该 worker |
| `handoff.main_known_missing_params` | 是 | main 已确认缺失的字段，可为空 |
| `output_contract` | 是 | 声明 worker 返回结构化结果，不直接拥有最终回答权 |

第三阶段预留但不实现完整能力：

| 字段 | 第三阶段取值 | 第四阶段目标 |
| --- | --- | --- |
| `context_package.retrieved_context` | `[]` | conversation_record 向量召回结果 |
| `context_package.long_term_memory` | `[]` | 用户长期记忆召回结果 |
| `context_package.memory_palace_refs` | `[]` | 记忆宫殿检索结果 |

说明：

- 第三阶段没有真正的 LLM 压缩摘要。
- 第三阶段的 `conversation_summary` 只是现有轻量滚动摘要。
- 第三阶段只保证几轮对话、用户原始消息、任务描述和 main 的自然语言交接信息可以进入 worker。
- main 给出的 `handoff_context` 只是辅助理解，不是 worker 的领域参数来源。
- worker 不能假设 `handoff_context` 完整，必须自己结合 `user_message`、`conversation_summary`、`recent_turns` 和 `previous_worker_result` 判断。
- 第四阶段负责把检索内容、压缩摘要、长期记忆和记忆宫殿结果全部接入该任务包。

### 7.3 传递边界

必须传递：

- 当前用户原始消息。
- main agent 理解后的任务目标。
- 当前 session 的轻量摘要。
- 相关最近几轮对话。
- 输出契约。

按需传递：

- 上一次同 worker 的执行结果。
- 当前 pending action。
- main 已理解到的自然语言关键信息。
- main 已确认缺失的字段。
- 和本任务直接相关的约束。

不要传递：

- main agent 完整 Prompt。
- main agent 完整 Agent 资产。
- 其他 worker 原始内部结果。
- API Key。
- 全量历史对话。
- 全量长期记忆。
- 数据库 ORM 原始对象。

### 7.4 兼容字段

兼容字段：

```json
{
  "action_detail": {
    "name": "amap_worker"
  }
}
```

第三阶段推荐统一使用 `worker_id`。

### 7.5 worker 返回结构

worker 返回给 main agent：

```json
{
  "status": "success",
  "data": {
    "worker_id": "amap_worker",
    "domain_result": {},
    "used_skills": []
  },
  "summary": "高德 worker 已完成路线和天气查询",
  "missing_params": [],
  "error": null
}
```

如果 worker 需要用户补充：

```json
{
  "status": "missing_params",
  "question": "请补充出发地。",
  "missing_params": ["origin_address"],
  "summary": "缺少出发地"
}
```

main agent 收到 worker 的缺参结果后，应回到 main Harness 和 main Loop 判断，再由 main 决定继续调用 worker 或追问用户。

### 7.6 缺参回流规则

worker 返回 `missing_params` 后，main agent 不应立刻机械追问。

main Loop 需要先判断：

```text
main 是否可以从已有上下文补齐？
```

如果可以补齐：

```text
main
  → 重新 call_agent(amap_worker)
  → 带上用户原始消息、轻量摘要、recent_turns、previous_worker_result
  → worker 继续执行
```

如果不能补齐：

```text
main
  → ask_user
  → awaiting_user
```

用户补充后：

```text
awaiting_user
  → main 重新组装 Worker Task Package
  → call_agent(amap_worker)
  → worker 继续执行
```

说明：

- worker 负责指出领域缺口。
- main 负责判断缺口是否能由已有上下文补齐。
- main 负责向用户追问。
- worker 不直接拥有最终用户交互权。

### 7.7 worker 短程任务状态

第三阶段 worker 可以拥有自己的短程任务状态。

短程任务状态包括：

- worker 已识别出的领域参数。
- worker 缺失的领域参数。
- worker 已调用的 Skill。
- Skill 返回的结构化结果。
- worker 对当前领域任务的完成状态。

worker 第三阶段不拥有：

- 独立长期记忆。
- 独立向量检索。
- 独立 LLM 压缩摘要。
- 独立记忆宫殿召回。

一个 main session 中的一次领域任务，可以对应 worker 的一次 worker task session。

worker task session 隶属于 main session。

当 main session 中该事项完成后，worker 的短程任务状态也完成。

## 八、Runtime 改造

### 8.1 AgentLoader

第三阶段需要支持加载：

```text
main agent
worker agent
```

建议扩展：

```python
load_main_agent() -> AgentDefinition
load_worker_agent(worker_id: str) -> AgentDefinition
load_workers() -> dict[str, AgentDefinition]
```

`AgentDefinition` 继续复用第二阶段结构：

```python
agent_id
files
skills
skill_registry
```

worker agent 与 main agent 的文件结构一致。

### 8.2 WorkerRegistry

WorkerRegistry 负责：

- 保存 worker_id 到 AgentDefinition 的映射。
- 向 main LoopDecider 注入可用 worker 摘要。
- 校验 `worker_id` 是否存在。

Prompt 注入摘要建议：

```json
{
  "worker_id": "amap_worker",
  "name": "高德地图 Worker",
  "description": "处理地址解析、天气查询、路径规划、行政区域、IP 定位和坐标转换。"
}
```

### 8.3 WorkerExecutor

WorkerExecutor 负责：

1. 解析 `worker_id`。
2. 校验 Worker Task Package。
3. 加载 worker AgentDefinition。
4. 构建 worker RuntimeRequest。
5. 将 `task`、`handoff_context`、`context_package`、`handoff` 和 `output_contract` 注入 worker 上下文。
6. 启动 worker 内部 Loop。
7. 收集 worker 的最终结果。
8. 转换为 main agent 的 ActionResult。

WorkerExecutor 不直接选择 Skill。

Skill 选择仍然发生在 worker LoopDecider。

领域参数抽取和缺参判断仍然发生在 worker LoopDecider 或 worker 内部 Skill 调用结果中。

WorkerExecutor 不负责检索长期记忆。

WorkerExecutor 只消费 ContextAssembler 已经放入 Worker Task Package 的上下文。

第三阶段 `retrieved_context`、`long_term_memory`、`memory_palace_refs` 可以为空数组。

### 8.4 RuntimeEngine

第三阶段 RuntimeEngine 需要支持嵌套运行：

```text
main RuntimeEngine
  └── WorkerExecutor
      └── worker RuntimeEngine
```

必须防止无限递归。

建议限制：

```text
main 可以 call_agent
worker 第一版不允许再 call_agent
```

worker 内部允许：

```text
answer_user
ask_user
call_skill
```

worker 内部暂不允许：

```text
call_agent
```

### 8.5 State 与记录

第三阶段可以继续复用第一、二阶段表：

```text
session_state
agent_run
agent_run_step
tool_call
```

记录建议：

- main run 记录 `call_agent(amap_worker)`。
- worker run 也创建自己的 `agent_run`。
- worker run 的 `agent_id = amap_worker`。
- `tool_call.tool_name` 对 main 来说是 `amap_worker`。
- worker 内部 Skill 的 `tool_call.tool_name` 是真实 skill_id。
- Worker Task Package 应写入 main `agent_run_step.action_detail`，方便排查 main 给 worker 的上下文边界。
- worker 内部 run 不保存 main 全量上下文，只保存本次任务包中传入的必要字段。

后续如果需要关联 main run 和 worker run，可以在技术债务中追加 `parent_run_id`。

## 九、AG-UI 中间步骤

第三阶段 AG-UI 需要展示两级步骤，但第一版可以保持扁平展示。

建议事件顺序：

```text
正在理解请求

正在调用 worker · amap_worker
worker 正在解析地址 · amap_geocode
worker 已拿到 amap_geocode 的结果
worker 正在查询路线 · amap_route_driving
worker 已拿到 amap_route_driving 的结果
worker 正在查询天气 · amap_weather
worker 已拿到 amap_weather 的结果
已拿到 amap_worker 的结果
正在生成最终回答
已完成
```

UI 不需要展示内部 Prompt。

UI 不展示 API Key。

UI 不展示原始接口响应全量 JSON。

## 十、迁移策略

6. 实现 WorkerLoader 和 WorkerRegistry。

## 十一、验收标准

第三阶段通过条件：

- main agent 可以扫描到 `amap_worker`。
- main agent 的 `call_agent` 可以调用真实 worker。
- worker agent 可以扫描自己的 Skill。
- 高德相关 Skill 不再由 main agent 直接持有。
- worker 可以调用 `api / chain / code` Skill。
- main 可以向 worker 传递 Worker Task Package。
- Worker Task Package 包含当前请求、任务、自然语言交接信息、轻量摘要和最近几轮对话拼接位置。
- 第三阶段预留的检索字段可以为空，但字段位置必须存在。
- worker 的 Skill 结果进入 worker Harness。
- worker 结果返回 main agent 后，main Harness 可以继续评估。
- 最终回答仍由 main agent 生成。
- 缺参时可以正确追问，并在用户补充后继续 worker 链路。
- AG-UI 可以展示 worker 调用和 worker 内部 Skill 步骤。
- 用户侧输出不暴露 Runtime、Prompt、内部 State、API Key 或原始异常堆栈。

## 十二、技术债务

第三阶段暂缓事项：

- `parent_run_id` 或 run tree 结构。
- worker 间调用。
- 多 worker 并行。
- worker 权限和配额。
- worker 独立部署。
- worker 自主管理 Skill。
- Agent 自动创建 worker。
- Skill 自动安装和变更审计。

这些内容进入后续阶段或 `docs/技术方案/技术债务.md`。
