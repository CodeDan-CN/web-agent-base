# 第三阶段详细架构设计

## 概述

本文档描述第三阶段「Worker Agent 闭环」的实施架构。

第一阶段已经跑通主 Agent 的 Loop 骨架。

第二阶段已经跑通主 Agent 调用真实 Skill 的闭环。

第三阶段的目标是把 `call_agent` 从 mock worker executor 替换为真实 worker agent 运行机制。

第三阶段完成后，main agent 不再直接持有高德领域 Skill，而是根据用户意图调用 `amap_worker`。`amap_worker` 拥有自己的 Agent 资产、SkillCatalog、Harness 和输出规范，并在 worker 内部调用高德基础 API Skill，最后把结构化结果返回 main agent，由 main agent 面向用户生成最终回答。

## 一、第三阶段范围

### 要实现

第三阶段需要实现：

- `agents/workers/` 目录扫描。
- Worker AgentDefinition 加载。
- Worker Agent 拥有独立 Markdown 资产。
- Worker Agent 拥有独立 `skills/` 目录和 SkillRegistry。
- main agent 的 `call_agent` 使用真实 `worker_id` 和 Worker Task Package 调用 worker。
- Worker Task Package 可以传递当前请求、任务描述、自然语言交接信息、轻量会话摘要和最近几轮对话。
- 为后续检索上下文、长期记忆或外部记忆服务保留兼容字段，但第三阶段不实现。
- worker 内部复用第二阶段 SkillExecutor。
- worker 内部执行自己的 LoopDecider、ActionExecutor、HarnessEvaluator 和 StateManager。
- worker 结果以结构化 `ActionResult` 返回 main agent。
- main agent Harness 评估 worker 返回结果。
- `tool_call` 记录真实 worker 调用。
- AG-UI 输出真实 worker 调用步骤。
- 高德相关 Skill 从 `agents/main/skills/` 下沉到 `agents/workers/amap_worker/skills/`。
- main / worker 在进入 Loop 前执行事件上下文预加载。
- 如果请求明确归属已存在事件，直接把该事件内与当前 agent 相关的一轮或多轮对话拼接进 Prompt。
- 如果当前会话中存在可能相关的历史事件，由模型做轻量相关性判断，命中后把对应事件内的一轮或多轮对话拼接进 Prompt。
- 事件结束后通过简易 Hook 沉淀 main / worker 各自的事件 summary。

### 不实现

第三阶段暂不实现：

- worker 自主创建 Skill。
- worker 自主创建其他 worker。
- worker 间任意互相调用。
- 多 worker 并行调度。
- Worker 权限系统。
- Worker 资源隔离。
- Worker 独立部署。
- Worker 独立数据库。
- Agent 市场或 Skill 市场。
- 记忆闭环。
- 记忆宫殿管理接口。
- 完整 LLM 对话压缩摘要体系。
- 全局跨会话事件检索。
- 长期记忆召回。
- 记忆宫殿召回。

第三阶段只解决 main agent 调用单个真实 worker agent 的闭环。

第三阶段同时补齐事件级上下文的最小闭环：

- 当前请求已有 `event_id` 时，按 `event_id + agent_id` 直接读取，并只拼接命中事件内的一轮或多轮对话。
- 当前会话存在历史事件但本轮未明确绑定事件时，可以先让模型判断这些历史事件是否与本轮请求相关。
- 命中相关事件时，只把命中事件内与当前 agent 相关的一轮或多轮对话注入 Prompt。
- 没有历史事件或判断不相关时，不注入事件内对话。

会话上下文压缩、recent turns 与 event dialogues 去重进入第四阶段。

长期记忆、记忆宫殿和跨会话检索不进入第四阶段，记录到技术债务。

## 二、目标链路

第三阶段需要验证以下链路：

```text
new_request
  → call_agent(amap_worker)
  → worker.new_request
  → worker.call_skill
  → worker.ready_to_plan
  → worker.answer_user
  → main.ready_to_plan
  → main.answer_user
  → completed
```

缺参链路：

```text
new_request
  → call_agent(amap_worker)
  → worker.missing_params
  → main.harness
  → main.loop
  → main.ask_user
  → awaiting_user
```

多轮补充链路：

```text
new_request
  → call_agent(amap_worker)
  → worker.missing_params
  → main.ask_user
  → awaiting_user

awaiting_user
  → call_agent(amap_worker)
  → worker.call_skill
  → worker.ready_to_plan
  → worker.answer_user
  → main.ready_to_plan
  → main.answer_user
  → completed
```

说明：

- main agent 只决定是否调用 worker，不直接理解高德 API 细节。
- 只要 main agent 已经能判断请求属于某个 worker 的领域，第一轮应优先进入该 worker，由 worker 判断领域缺参。
- worker agent 负责在自己的领域内选择 Skill。
- worker agent 负责判断领域任务是否缺少必要信息。
- worker 的 `answer_user` 在第三阶段不直接面向最终用户，而是生成给 main agent 的结构化领域结果。
- 最终用户回答仍由 main agent 输出。

main agent 保留：

```text
agents/main/skills/content_extract/
```

```text
agents/main/skills/amap_geocode/
```

- main agent 不直接拼接高德 API 参数。
- main agent 不直接调用高德 Skill。

## 五、amap_worker 设计

### 5.1 Worker 定位

`amap_worker` 是高德 Web 服务领域 worker。

职责：

- 理解用户的地图、地址、路线、天气、行政区划、IP 定位、坐标转换诉求。
- 判断是否需要追问。
- 在自己的 SkillCatalog 中选择合适 Skill。
- 将 Skill 执行结果组织成结构化领域结果。
- 不直接输出最终用户回答的完整文案。

### 5.2 Worker 资产

`amap_worker` 需要包含：

```text
agents/workers/amap_worker/
├── SOUL.md
├── Agent.md
├── Instruction.md
├── tools.md
├── output.md
├── harness.md
├── prompts/
│   ├── fragments.md
│   └── examples.md
└── skills/
```

各文件职责与 main agent 保持一致。

内容差异：

- `Agent.md` 描述高德领域职责和边界。
- `Instruction.md` 描述高德任务处理 SOP。
- `tools.md` 描述 worker 内部 Skill 使用规则。
- `output.md` 要求 worker 输出结构化领域结果，不面向最终用户发挥。
- `harness.md` 评估高德 Skill 结果是否足以返回 main agent。

## 七、Worker Task Package 与上下文边界

Worker Task Package 是 main agent 派发给 worker 的轻量任务包。

它解决的问题不是 worker 如何执行 Skill，而是 main agent 传给 worker 什么、不给什么、边界在哪里。

main agent 不负责一次性给齐 worker 需要的所有领域参数。

worker 必须基于任务包自行完成领域参数抽取、缺参判断、Skill 选择和结构化结果组织。

第三阶段必须先把这个协议跑通。

第四阶段再补齐会话上下文压缩、recent turns 与 event dialogues 去重。

### 7.1 Main LoopDecider 输出结构

main agent 调用 worker 时，LoopDecider 只输出轻量 `action_detail`。

LoopDecider 不负责拼接 `context_package`。

LoopDecider 不负责补齐 `output_contract`。

LoopDecider 不负责填写检索、长期记忆或外部记忆服务字段。

```json
{
  "action": "call_agent",
  "action_detail": {
    "worker_id": "amap_worker",
    "task": "规划路线并查看目的地天气",
    "handoff_context": "用户从杭州市西湖区文三路出发，目的地是杭州市上城区湖滨银泰，希望开车过去，并查看目的地天气。",
    "handoff": {
      "reason": "请求属于高德地图领域"
    }
  },
  "reason": "需要高德地图 Worker 处理"
}
```

### 7.2 Runtime 补齐后的 Worker Task Package

Runtime 看到 `call_agent` 后，基于轻量 `action_detail` 补齐标准 Worker Task Package。

补齐发生在 Runtime / ContextAssembler / WorkerExecutor。

完整任务包示例：

```json
{
  "worker_id": "amap_worker",
  "task": "规划路线并查看目的地天气",
  "handoff_context": "用户从杭州市西湖区文三路出发，目的地是杭州市上城区湖滨银泰，希望开车过去，并查看目的地天气。",
  "handoff": {
    "reason": "请求属于高德地图领域"
  },
  "context_package": {
    "event_id": "当前主事件 ID，第三阶段由 Runtime 生成或复用",
    "user_message": "用户原始请求",
    "conversation_summary": "当前 session 的轻量滚动摘要",
    "event_history": [],
    "recent_turns": [],
    "retrieved_context": [],
    "long_term_memory": [],
    "memory_palace_refs": [],
    "previous_worker_result": null
  },
  "output_contract": {
    "format": "structured_result",
    "final_answer_owner": "main_agent"
  }
}
```

### 7.3 字段来源

第三阶段实际传递：

| 字段 | 来源 | 第三阶段说明 |
| --- | --- | --- |
| `worker_id` | Main LoopDecider | 目标 worker |
| `task` | Main LoopDecider | 简短任务描述，不要求包含全部领域参数 |
| `handoff_context` | Main LoopDecider | main 已理解到的自然语言交接信息，可为空 |
| `handoff.reason` | Main LoopDecider | main 为什么把任务交给该 worker |
| `context_package.event_id` | Runtime | 当前主事件 ID，worker 使用同一事件，不创建独立事件 |
| `context_package.user_message` | Runtime | 当前用户原始消息 |
| `context_package.conversation_summary` | Runtime | 当前 `session_state.conversation_summary`，只是轻量滚动摘要 |
| `context_package.event_history` | Runtime | 同一事件下 main / worker 的运行和工具调用历史，第三阶段可以先为空 |
| `context_package.recent_turns` | Runtime | 最近几轮对话的拼接位置，第三阶段可以先从现有 run/session 信息构造简版 |
| `context_package.previous_worker_result` | Runtime | 上一次相关 worker 结果，可为空 |
| `output_contract` | Runtime | 声明 worker 返回结构化结果，不直接拥有最终回答权 |

第三阶段预留但不实现完整能力：

| 字段 | 第三阶段取值 | 后续目标 |
| --- | --- | --- |
| `context_package.retrieved_context` | `[]` | 后续外部检索上下文 |
| `context_package.long_term_memory` | `[]` | 后续用户长期记忆召回结果 |
| `context_package.memory_palace_refs` | `[]` | 后续个人助手型记忆宫殿结果 |

说明：

- 第三阶段没有真正的 LLM 压缩摘要。
- 第三阶段的 `conversation_summary` 只是现有轻量滚动摘要。
- 第三阶段只保证几轮对话、用户原始消息、任务描述和 main 的自然语言交接信息可以进入 worker。
- main 给出的 `handoff_context` 只是辅助理解，不是 worker 的领域参数来源。
- worker 不能假设 `handoff_context` 完整，必须自己结合 `user_message`、`event_history`、`conversation_summary`、`recent_turns` 和 `previous_worker_result` 判断。
- 第四阶段负责会话上下文压缩和去重，不负责把长期记忆和记忆宫殿结果接入该任务包。

worker 不创建独立事件。

main 调用 worker 时，worker 运行使用 main 当前 `event_id`。worker 的 run、step 和 Skill tool_call 都归属同一个事件。

这能改善同一事件内 worker 忘记前面执行路径的问题，例如已经解析过起点、终点之后，后续步骤可以通过 `event_history` 看到已经拿到的坐标，再继续路线或天气查询。

### 7.4 Loop 前事件上下文预加载

main agent 和 worker agent 在开始自己的 Loop 前，都需要先执行事件上下文预加载。

预加载发生在 Runtime 已经拿到 `user_id / session_id / agent_id / event_id` 之后，LoopDecider 调用之前。

它的职责是把“当前这件事”相关的历史内容拼接进 Prompt，而不是决定下一步 Action。

执行顺序：

```text
RuntimeRequest 进入
  ↓
EventManager 解析或创建当前 event
  ↓
EventContextPreloader 检查事件上下文
  ↓
ContextAssembler 组装 Prompt
  ↓
LoopDecider 判断下一步 Action
```

#### 7.4.1 已有事件直接拼接

如果本轮请求已经带有 `event_id`，并且数据库中可以查询到该事件，则不需要让模型判断相关性。

Runtime 直接读取：

- `agent_event(event_id)`。
- 当前 `event_id + agent_id` 对应的 `agent_event_context.summary`。
- 当前 `event_id` 下该 agent 已有的对话记录。
- 当前 `event_id` 下该 agent 的必要 run / step / tool_call 摘要。

其中 `event_id`、事件 summary 和执行摘要只作为内部查询、筛选和排序依据。

真正注入 Prompt 的内容只包含：当前 agent 在该事件内最相关的一轮或多轮对话。

原因是：既然事件已经明确存在，就说明这轮请求正在延续同一件事，模型不需要再次判断“是不是相关”。

#### 7.4.2 无明确事件时检查会话历史事件

如果本轮没有明确 `event_id`，但当前 `user_id + session_id` 下存在历史事件，则 Runtime 先取最近若干个候选事件。

候选事件建议只来自当前会话，不做跨会话全局检索。

候选内容只给模型最小信息：

```json
[
  {
    "event_id": "evt_xxx",
    "title": "杭州路线与天气",
    "summary": "用户查询过从文三路到湖滨银泰的驾车路线和目的地天气。",
    "status": "completed",
    "updated_at": "2026-06-25T10:00:00"
  }
]
```

模型只负责判断：

```json
{
  "related": true,
  "event_id": "evt_xxx",
  "confidence": "possible",
  "reason": "用户继续追问目的地停车，可能延续上一轮出行事件"
}
```

规则：

- 如果没有候选事件，跳过该步骤。
- 如果模型判断不相关，跳过事件内对话注入。
- 如果模型判断 `related = true`，只取一个最相关事件。
- 第一版允许 `confidence = possible` 也注入，因为用户追问常常表达很短。
- 该模型调用只用于事件相关性判断，不参与 Loop Action 决策。

#### 7.4.3 空事件不处理

如果当前会话没有任何历史事件，或者没有可查询到的事件上下文，则不执行事件相关性判断。

此时 Prompt 只包含当前请求、最近几轮对话、Agent 资产和可用 Skill / worker 摘要。

#### 7.4.4 注入内容边界

允许注入：

- 与当前 agent 相关的命中事件内一轮或多轮对话内容。

不允许注入：

- 命中事件的 `event_id`。
- 命中事件标题。
- 当前 agent 自己的 `agent_event_context.summary`。
- 同一事件下 worker / Skill 执行摘要。
- 全量历史事件列表。
- 全量原始 tool response。
- API Key。
- 其他用户事件。
- 其他会话的事件，第三阶段暂不做跨会话召回。
- main / worker 完整内部 Prompt。

#### 7.4.5 main 与 worker 的差异

main agent：

- 负责识别当前用户请求是否延续当前会话中的某个事件。
- 可以使用 main 自己的事件摘要、最近用户对话和 worker 调用摘要做内部判断。
- 注入 Prompt 时只保留命中事件内与 main 相关的一轮或多轮对话。
- 仍然由 LoopDecider 判断是否继续 `call_agent`、`ask_user` 或 `answer_user`。

worker agent：

- 不重新判断事件归属。
- 使用 main 传入的 `event_id`。
- 读取同一 `event_id + worker_agent_id` 下自己的上下文。
- 可以额外接收 main 放入 Worker Task Package 的 `event_history`，用于内部理解当前主事件已经发生过什么。
- 注入 Prompt 时只保留命中事件内与 worker 相关的一轮或多轮对话。

这样可以保证：用户对同一件事继续追问时，main 能拿回前序事件；worker 被再次调用时，也能知道自己在该事件中之前已经执行过哪些领域步骤。

### 7.5 传递边界

必须传递：

- 当前用户原始消息。
- 当前事件 ID。
- 同一事件内 main / worker 的必要执行历史。
- main agent 理解后的任务目标。
- 当前 session 的轻量摘要。
- 相关最近几轮对话。
- 输出契约。

按需传递：

- 上一次同 worker 的执行结果。
- 当前 pending action。
- main 已理解到的自然语言关键信息。
- main 已确认缺失的字段。
- 和本任务直接相关的约束。

不要传递：

- main agent 完整 Prompt。
- main agent 完整 Agent 资产。
- 其他 worker 原始内部结果。
- API Key。
- 全量历史对话。
- 全量长期记忆。
- 数据库 ORM 原始对象。

### 7.6 兼容字段

兼容字段：

```json
{
  "action_detail": {
    "name": "amap_worker"
  }
}
```

第三阶段推荐统一使用 `worker_id`。

### 7.7 worker 返回结构

worker 返回给 main agent：

```json
{
  "status": "success",
  "data": {
    "worker_id": "amap_worker",
    "domain_result": {},
    "used_skills": []
  },
  "summary": "高德 worker 已完成路线和天气查询",
  "missing_params": [],
  "error": null
}
```

如果 worker 需要用户补充：

```json
{
  "status": "missing_params",
  "question": "请补充出发地。",
  "missing_params": ["origin_address"],
  "summary": "缺少出发地"
}
```

main agent 收到 worker 的缺参结果后，应回到 main Harness 和 main Loop 判断，再由 main 决定继续调用 worker 或追问用户。

### 7.8 缺参回流规则

worker 返回 `missing_params` 后，main agent 不应立刻机械追问。

main Loop 需要先判断：

```text
main 是否可以从已有上下文补齐？
```

如果可以补齐：

```text
main
  → 重新 call_agent(amap_worker)
  → 带上用户原始消息、轻量摘要、recent_turns、previous_worker_result
  → worker 继续执行
```

如果不能补齐：

```text
main
  → ask_user
  → awaiting_user
```

用户补充后：

```text
awaiting_user
  → main 重新组装 Worker Task Package
  → call_agent(amap_worker)
  → worker 继续执行
```

说明：

- worker 负责指出领域缺口。
- main 负责判断缺口是否能由已有上下文补齐。
- main 负责向用户追问。
- worker 不直接拥有最终用户交互权。

### 7.9 worker 短程任务状态

第三阶段 worker 可以拥有自己的短程任务状态。

短程任务状态包括：

- worker 已识别出的领域参数。
- worker 缺失的领域参数。
- worker 已调用的 Skill。
- Skill 返回的结构化结果。
- worker 对当前领域任务的完成状态。

worker 第三阶段不拥有：

- 独立长期记忆。
- 独立跨事件检索。
- 独立 LLM 压缩摘要。
- 独立记忆宫殿召回。

一个 main session 中的一次领域任务，可以对应 worker 的一次 worker task session。

worker task session 隶属于 main session。

当 main session 中该事项完成后，worker 的短程任务状态也完成。

## 八、Runtime 改造

### 8.1 AgentLoader

第三阶段需要支持加载：

```text
main agent
worker agent
```

建议扩展：

```python
load_main_agent() -> AgentDefinition
load_worker_agent(worker_id: str) -> AgentDefinition
load_workers() -> dict[str, AgentDefinition]
```

`AgentDefinition` 继续复用第二阶段结构：

```python
agent_id
files
skills
skill_registry
```

worker agent 与 main agent 的文件结构一致。

### 8.2 WorkerRegistry

WorkerRegistry 负责：

- 保存 worker_id 到 AgentDefinition 的映射。
- 向 main LoopDecider 注入可用 worker 摘要。
- 校验 `worker_id` 是否存在。

Prompt 注入摘要建议：

```json
{
  "worker_id": "amap_worker",
  "name": "高德地图 Worker",
  "description": "处理地址解析、天气查询、路径规划、行政区域、IP 定位和坐标转换。"
}
```

### 8.3 WorkerExecutor

WorkerExecutor 负责：

1. 解析 `worker_id`。
2. 接收 Main LoopDecider 输出的轻量 `action_detail`。
3. 从 Runtime 上下文补齐 `context_package` 和 `output_contract`。
4. 构造完整 Worker Task Package。
5. 校验 Worker Task Package。
6. 加载 worker AgentDefinition。
7. 构建 worker RuntimeRequest。
8. 将 Worker Task Package 注入 worker 上下文。
9. 启动 worker 内部 Loop。
10. 收集 worker 的最终结果。
11. 转换为 main agent 的 ActionResult。

WorkerExecutor 不直接选择 Skill。

Skill 选择仍然发生在 worker LoopDecider。

领域参数抽取和缺参判断仍然发生在 worker LoopDecider 或 worker 内部 Skill 调用结果中。

WorkerExecutor 不负责检索长期记忆。

WorkerExecutor 只消费 ContextAssembler 已经放入 Worker Task Package 的上下文。

第三阶段 `retrieved_context`、`long_term_memory`、`memory_palace_refs` 可以为空数组。

### 8.4 EventContextPreloader

EventContextPreloader 是第三阶段新增的 Loop 前置上下文装配器。

它运行在 ContextAssembler 之前，输出 `current_event_context`。

职责：

- 根据 `event_id` 直接读取当前事件上下文。
- 在没有明确 `event_id` 时，读取当前会话候选历史事件。
- 必要时调用模型判断本轮请求是否延续候选事件。
- 将命中事件内与当前 agent 相关的一轮或多轮对话交给 ContextAssembler。

它不负责：

- 创建最终回答。
- 判断 Loop Action。
- 调用 Skill。
- 调用 worker。
- 写长期记忆。
- 调用记忆宫殿。

建议接口：

```python
class EventContextPreloader:
    async def preload(
        self,
        user_id: str,
        session_id: str,
        agent_id: str,
        event_id: str | None,
        user_message: str,
    ) -> EventPromptContext:
        ...
```

输出结构建议：

```json
{
  "matched_event_id": "evt_xxx",
  "source": "explicit_event",
  "event_dialogues": [],
  "should_inject": true
}
```

`matched_event_id` 和 `source` 只用于 Runtime 内部追踪，不直接注入 Prompt。

Prompt 中只拼接 `event_dialogues`。

`source` 可选：

```text
explicit_event
related_session_event
empty
```

### 8.5 RuntimeEngine

第三阶段 RuntimeEngine 需要支持嵌套运行：

```text
main RuntimeEngine
  └── WorkerExecutor
      └── worker RuntimeEngine
```

必须防止无限递归。

建议限制：

```text
main 可以 call_agent
worker 第一版不允许再 call_agent
```

worker 内部允许：

```text
answer_user
ask_user
call_skill
```

worker 内部暂不允许：

```text
call_agent
```

### 8.6 State 与记录

第三阶段可以继续复用第一、二阶段表：

```text
session_state
agent_run
agent_run_step
tool_call
```

记录建议：

- main run 记录 `call_agent(amap_worker)`。
- worker run 也创建自己的 `agent_run`。
- worker run 的 `agent_id = amap_worker`。
- `tool_call.tool_name` 对 main 来说是 `amap_worker`。
- worker 内部 Skill 的 `tool_call.tool_name` 是真实 skill_id。
- Main LoopDecider 的轻量 `action_detail` 写入 main `agent_run_step.action_detail`。
- Runtime 补齐后的完整 Worker Task Package 写入 main `tool_call.input_payload` 或 worker run 输入摘要，方便排查 main 给 worker 的上下文边界。
- worker 内部 run 不保存 main 全量上下文，只保存本次任务包中传入的必要字段。

后续如果需要关联 main run 和 worker run，可以在技术债务中追加 `parent_run_id`。

## 十、事件 Summary 与简易 Hook

第三阶段开始引入事件级上下文的最小实现。

事件不是 Loop State。

事件用于描述“用户这一件事”，可以跨单轮或多轮请求。

main agent 和 worker agent 使用同一个 `event_id`。

worker 不创建独立事件。

worker 的 run、step、tool_call 都通过 Runtime 关联到 main 当前事件。

### 10.1 事件 Summary

同一个事件下，不同 agent 拥有自己的 summary。

```text
agent_event
  ├── agent_event_context(event_id, agent_id = main)
  └── agent_event_context(event_id, agent_id = amap_worker)
```

原因：

- main 的历史是用户交互、worker 调用和最终回答。
- worker 的历史是领域参数抽取、Skill 调用和领域结果。
- 两者共享同一个事件，但上下文重点不同。

summary 生成规则：

```text
主 Agent Harness / Runtime 判断事件结束
  ↓
触发 EventSummaryRequested
  ↓
Hook 按 event_id + agent_id 拉取历史
  ↓
未超过轮数和字数限制：直接汇总
  ↓
超过限制：调用模型总结为约 100 字
  ↓
写入 agent_event_context.summary
```

默认限制：

```text
EVENT_SUMMARY_DIRECT_MAX_TURNS=6
EVENT_SUMMARY_DIRECT_MAX_CHARS=1200
EVENT_SUMMARY_TARGET_CHARS=100
```

### 10.2 Loop 前事件内对话注入

事件 summary 生成发生在事件结束后；事件内对话注入发生在下一次 Loop 开始前。

两者是配套关系：

```text
上一轮事件结束
  ↓
EventSummaryRequested Hook 生成 agent_event_context.summary
  ↓
下一轮请求进入
  ↓
EventContextPreloader 读取或匹配相关事件
  ↓
ContextAssembler 把命中事件内的一轮或多轮对话拼进 Prompt
  ↓
LoopDecider 基于更完整上下文判断下一步
```

如果请求已经带有存在的 `event_id`：

```text
直接读取该 event_id 对应上下文
不做模型相关性判断
Prompt 只注入该事件内与当前 agent 相关的一轮或多轮对话
```

如果请求没有明确 `event_id`，但当前会话存在历史事件：

```text
取当前会话最近候选事件
调用轻量模型判断是否相关或可能相关
相关则注入命中事件内与当前 agent 相关的一轮或多轮对话
不相关则跳过
```

如果当前会话没有历史事件：

```text
跳过事件内对话注入
```

第三阶段只做当前会话内的事件相关性判断，不做跨会话、跨用户或长期记忆检索。

### 10.3 简易 Hook 设计

第三阶段使用当前已有的进程内轻量 Hook。

新增事件：

```text
EventSummaryRequested
```

边界：

- 它不是 Loop Action。
- 它不是 Skill。
- 它不是 worker。
- 它不进入 Harness。
- 它不影响当前 Action 判断。
- 它不应该阻塞用户最终回答。

第一版实现：

```text
RuntimeEngine
  ↓
主事件 completed / failed
  ↓
asyncio.create_task(RuntimeHook.on_event_summary_requested)
  ↓
RuntimeHook 生成或更新 agent_event_context.summary
```

失败策略：

- Hook 内部捕获异常。
- 写日志。
- 不回滚 Agent Run。
- 不影响用户已经拿到的回答。

复杂队列化 Hook、重试、失败队列和后台 worker 仍然放在技术债务中，不进入第三阶段。

## 十二、验收标准

第三阶段通过条件：

- main agent 可以扫描到 `amap_worker`。
- main agent 的 `call_agent` 可以调用真实 worker。
- worker agent 可以扫描自己的 Skill。
- 高德相关 Skill 不再由 main agent 直接持有。
- worker 可以调用 `api / chain / code` Skill。
- main 可以向 worker 传递 Worker Task Package。
- Worker Task Package 包含当前请求、任务、自然语言交接信息、轻量摘要和最近几轮对话拼接位置。
- main 和 worker 可以共享同一个 `event_id`。
- 主事件结束后可以通过简易 Hook 沉淀 main / worker 各自的事件 summary。
- 下一轮请求进入 Loop 前可以读取已存在事件，并把该事件内与当前 agent 相关的一轮或多轮对话拼接进 Prompt。
- 当前会话有历史事件但本轮未明确绑定事件时，可以由模型判断是否相关，并只注入命中事件内的一轮或多轮对话。
- 当前会话无历史事件时，不执行事件相关性判断。
- 第三阶段预留的检索字段可以为空，但字段位置必须存在。
- worker 的 Skill 结果进入 worker Harness。
- worker 结果返回 main agent 后，main Harness 可以继续评估。
- 最终回答仍由 main agent 生成。
- 缺参时可以正确追问，并在用户补充后继续 worker 链路。
- AG-UI 可以展示 worker 调用和 worker 内部 Skill 步骤。
- 用户侧输出不暴露 Runtime、Prompt、内部 State、API Key 或原始异常堆栈。
