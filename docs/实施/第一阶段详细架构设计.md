# 第一阶段详细架构设计

## 概述

本文档描述第一阶段「Loop 全链路骨架」的实施架构。

第一阶段目标不是完成完整 Agent 能力，而是跑通 Agent 框架最核心的闭环：

```text
HTTP Request
  ↓
web
  ↓
RuntimeEngine
  ↓
ContextAssembler
  ↓
LoopDecider
  ↓
ActionExecutor
  ↓
HarnessEvaluator
  ↓
StateManager
  ↓
RuntimeHook
  ↓
HTTP Response
```

第一阶段必须覆盖全部合法 State / Action 分支，允许使用 mock executor 模拟 `call_skill` 和 `call_agent` 的外部返回。

参考文档：

| 文档 | 说明 |
| --- | --- |
| `../技术方案/脚手架设计.md` | 目录结构和工程边界 |
| `../技术方案/技术方案.md` | 总体技术方案 |
| `../PRD/Loop 设计优化版本.md` | Loop 核心规则 |
| `../技术方案/技术债务.md` | 后续演进事项 |

## 一、第一阶段范围

### 要实现

第一阶段需要实现：

- FastAPI 服务启动。
- `/health` 可访问。
- `/agent/run` 可调用。
- `/agent/run/stream` 可按 AG-UI 事件流输出。
- `/session/{session_id}` 可查询会话快照。
- Agent Markdown 文件可加载。
- main agent 可运行。
- mock skill executor 可被调用。
- mock agent executor 可被调用。
- State / Action 可完整流转。
- Harness 可评估 mock executor 结果。
- Runtime Hook 可接收基础事件。
- Runtime 中间步骤可转换为 AG-UI 事件。
- `session_state` 可保存。
- `agent_run` 可保存。
- `agent_run_step` 可追加写入。
- `tool_call` 可保存。

### 不实现

第一阶段暂不实现：

- `conversation_record` 完整写入。
- 对话 embedding。
- 对话向量召回。
- `MemoryPalaceProvider` 外部服务接入。
- `memory_manager` 子 Agent。
- ES / OpenSearch / ClickHouse。
- 外部消息队列。
- 真实 Skill 执行框架。
- 真实子 Agent 执行框架。
- 完整业务 Skill。
- 完整业务 worker。

第一阶段只保留必要接口和扩展点，不把第四阶段记忆能力提前做进去。

## 二、目标链路

第一阶段必须至少验证以下链路：

```text
new_request → answer_user → completed
new_request → ask_user → awaiting_user
awaiting_user → answer_user → completed

new_request → call_skill → ready_to_plan → answer_user → completed
new_request → call_skill → missing_params → ask_user → awaiting_user
new_request → call_skill → failed

new_request → call_agent → ready_to_plan → answer_user → completed
new_request → call_agent → missing_params → ask_user → awaiting_user
new_request → call_agent → failed

ready_to_plan → ask_user → awaiting_user
missing_params → call_skill → ready_to_plan
missing_params → call_agent → ready_to_plan
completed → end
failed → end
```

这些链路可以通过 mock skill executor / mock agent executor 触发。

注意：目标链路用于验证第一阶段状态机能力覆盖，不代表所有正常用户请求都必须强行命中这些链路。
当请求明显缺少必要信息，或依赖当前不可访问的外部资料时，`LLMLoopDecider` 可以直接选择 `ask_user` 并进入 `awaiting_user`。
这属于正常产品行为。
多轮用例的通过判定应重点看用户补充信息后，是否能回到正确的 `call_skill` 或 `call_agent` 链路并最终完成。

## 四、核心模块

### 4.1 app.py

职责：

- 创建 FastAPI app。
- 注册 web 路由。
- 注册异常处理器。
- 初始化 Tortoise ORM。
- 注册关闭事件。

`app.py` 不写业务逻辑。

### 4.2 web

职责：

- 接收 HTTP 请求。
- 校验 Pydantic DTO。
- 调用 `RuntimeEngine.run()`。
- 返回统一响应。

第一阶段接口：

```text
GET /health
POST /agent/run
POST /agent/run/stream
GET /session/{session_id}
```

`web/` 不直接调用 mock executor、Harness。

### 4.3 schema/api

第一阶段 API DTO：

```text
AgentRunRequest
AgentRunResponse
SessionStateResponse
BaseResponse
```

`AgentRunRequest` 字段：

```text
user_id
agent_id
session_id
request_id
message
metadata
```

不引入 `tenant_id`。

### 4.4 schema/db

第一阶段 DB Model：

```text
SessionState
AgentRun
AgentRunStep
ToolCall
```

第一阶段不实现：

```text
UserLongTermMemory
ConversationRecord
```

这两个进入第四阶段记忆闭环。

### 4.5 runtime/engine

`RuntimeEngine` 是第一阶段主编排器。

职责：

- 创建 `agent_run`。
- 读取或创建 `session_state`。
- 驱动 Loop。
- 执行 Action。
- 调用 Harness。
- 触发 Hook 事件。
- 构建最终响应。

核心流程：

```text
load_or_create_session
  ↓
create_agent_run
  ↓
while state not terminal
  ↓
assemble_context
  ↓
decide_action
  ↓
execute_action
  ↓
evaluate_if_needed
  ↓
next_state
  ↓
emit LoopStepCompleted
  ↓
flush session_state if boundary
  ↓
build_response
```

### 4.6 runtime/agents

职责：

- 根据目录约定加载 Agent。
- 读取 Markdown 文件。
- 形成 `AgentDefinition`。
- 第一阶段只加载 main agent。
- 真实 worker Agent 扫描和真实 Skill 扫描留到后续阶段。

第一阶段最低要求：

- 能加载 `agents/main/`。
- 缺失必要 Markdown 文件时明确报错。

不使用 `agent.yaml`。

### 4.7 runtime/context

职责：

- 组装 Loop Prompt Context。
- 组装 Answer Prompt Context。
- 组装 Harness Prompt Context。
- 组装 mock executor 调用所需的 `session_context`。

第一阶段 Context 来源：

```text
AgentDefinition.files
User request
SessionState
Previous action result
Harness feedback
Output schema
```

第一阶段多轮会话需要提供轻量 `conversation_summary`。

该摘要只服务于当前 Runtime 的多轮链路测试，用于表达上一轮意图、缺失参数和已补充信息。

不要求实现完整对话表、embedding 或语义召回。

第一阶段不接入：

```text
long_term_memory
conversation semantic retrieval
MemoryPalaceProvider
```

这些来源保留接口，返回空结果即可。

### 4.8 runtime/loop

职责：

- 定义 State。
- 定义 Action。
- 调用大模型判断下一步 Action。
- 校验 Action 输出结构。

第一阶段需要实现 `LLMLoopDecider`。

mock skill executor 和 mock agent executor 只提供可预测的外部返回，不控制 Loop 分支。

Loop 是否选择 `answer_user / ask_user / call_skill / call_agent`，必须由 `LLMLoopDecider` 根据用户问题、Agent 文件、工具说明和上下文判断。

### 4.9 runtime/actions

职责：

- 根据 Action 执行动作。

Action Executor 需要支持：

| Action | 第一阶段处理 |
| --- | --- |
| `answer_user` | 返回模型 answer |
| `ask_user` | 保存 pending_action，返回 question |
| `call_skill` | 调用 mock skill executor |
| `call_agent` | 调用 mock agent executor |

`answer_user` 和 `ask_user` 不进入 Harness。

`call_skill` 和 `call_agent` 必须进入 Harness。

执行 `call_skill` / `call_agent` 时，ActionExecutor 需要把本轮 Action 参数和 ContextAssembler 生成的 `session_context` 一起传给 mock executor。

### 4.10 runtime/harness

职责：

- 通过大模型评估 mock executor 结果。
- 输出下一步 State。

第一阶段 Harness 输出：

```text
ready_to_plan
missing_params
failed
```

第一阶段 Harness 必须真实落地为 LLM Harness。

不能只用规则把 `result.status` 映射成 State。

mock executor 只负责制造可控输入，Harness 必须读取执行结果、用户请求、当前 state、pending_action、missing_params 和 `harness.md` 规则后，由大模型判断结果是否可用。

Harness 输出必须结构化：

```json
{
  "state": "ready_to_plan",
  "status": "success",
  "summary": "content_extract_skill 返回的信息可以支持继续规划",
  "missing_params": [],
  "reason": "结果完整且没有阻塞项"
}
```

第一阶段可以让 mock executor 返回足够明确，但最终判断仍由 LLM Harness 给出。

### 4.11 utils/llm_client

职责：

- 统一封装模型调用。
- 屏蔽具体模型服务商差异。
- 为 `LLMLoopDecider`、`answer_user`、`LLM Harness`、`LLM Case Judge` 提供同一套调用入口。
- 作为 `utils/` 下的公共基础能力，避免 Runtime、Harness、测试判定器各自重复封装模型客户端。

第一阶段模型调用统一使用 OpenAI-compatible 方式。

实现建议：

- 使用 `openai` Python SDK。
- 通过 `.env` 注入 `MODEL_BASE_URL`、`MODEL_API_KEY`、`MODEL_NAME`。
- SDK 初始化时设置 `base_url`，不在业务代码中硬编码模型服务地址。
- 所有模型调用都经过 `utils/llm_client`，其他模块不直接调用 SDK。
- 模型返回必须先解析成项目内部结构，再交给 Loop、Harness 或 Answer 逻辑使用。

调用用途：

| 调用方 | 用途 |
| --- | --- |
| `LLMLoopDecider` | 判断下一步 Action |
| `answer_user` | 生成最终回答或追问 |
| `LLM Harness` | 评估 mock executor 结果 |
| `LLM Case Judge` | 判断测试用例是否完成预期链路 |

当前模型服务已通过 OpenAI-compatible `/chat/completions` 最小探测：

```text
base_url: https://token-plan-cn.xiaomimimo.com/v1
model: mimo-v2.5-pro
response.object: chat.completion
```

因此第一阶段可以按 OpenAI SDK 兼容模式接入。

### 4.12 runtime/state

职责：

- 读取或创建 `session_state`。
- 在内存中计算下一步 State。
- 更新 loop_count。
- 判断 max_loop_steps。
- 仅在边界状态 flush `session_state`。

`session_state` 是快照表，不是日志表。

每轮状态流转写入 `agent_run_step`。

边界状态：

```text
awaiting_user
completed
failed
```

### 4.13 runtime/hooks

Runtime Hook 是第一阶段必须实现的轻量事件机制。

第一阶段事件：

| Event | 第一阶段处理 |
| --- | --- |
| `UserMessageReceived` | 记录事件，可先不写 conversation_record |
| `LoopStepCompleted` | 追加写入 agent_run_step |
| `ToolCallCompleted` | 写入 tool_call |
| `AssistantMessageGenerated` | 记录事件，可先不写 conversation_record |
| `SessionStateChanged` | 必要时更新 session_state |

第一阶段原则：

- Hook 不参与 Loop 决策。
- Hook 不修改 Action。
- Hook 失败不影响主响应。
- Hook 错误写入日志。
- 不引入外部队列。

`conversation_record` 和 embedding 在第四阶段补齐。

### 4.14 runtime/ag_ui

职责：

- 将 Runtime 内部事件转换为 AG-UI 事件。
- 为 `/agent/run/stream` 提供标准事件流。
- 屏蔽 Runtime 内部字段和 AG-UI 外部协议之间的差异。

第一阶段需要实现 `AGUIEventAdapter`。

`AGUIEventAdapter` 不参与 Loop 决策，不修改 Action，不替代 Runtime Hook。

Runtime Hook 负责持久化内部记录。

AG-UI Adapter 负责对外输出用户界面可消费的事件。

两者可以消费同一份 Runtime 内部事件，但职责必须分开。

## 五、数据库落地

第一阶段只建必要表：

```text
session_state
agent_run
agent_run_step
tool_call
```

### 5.1 session_state

用途：

- 保存当前会话快照。
- 支持 `awaiting_user` 后恢复会话。

写入策略：

- 请求开始时读取或创建。
- 中间 State 在内存中流转。
- 进入边界状态时更新。

### 5.2 agent_run

用途：

- 保存一次 `/agent/run` 请求的总体记录。
- 记录最终状态和最终回答。

写入策略：

- 请求开始时创建。
- 请求结束时更新 final_state / final_answer / error_message。

### 5.3 agent_run_step

用途：

- 保存每轮 Loop 状态流转日志。

写入策略：

- append-only。
- 每轮追加一条。
- 不更新历史记录。

记录内容：

```text
state_before
action
action_detail
execution_result
harness_feedback
state_after
```

### 5.4 tool_call

用途：

- 保存 mock executor 调用记录。

写入策略：

- `call_skill` 和 `call_agent` 完成后追加。
- 由 `ToolCallCompleted` Hook 处理。

## 六、API 设计

### 6.1 GET /health

用于服务健康检查。

响应：

```json
{
  "status": "ok"
}
```

### 6.2 POST /agent/run

核心运行接口。

请求：

```json
{
  "user_id": "user_001",
  "agent_id": "main",
  "session_id": null,
  "request_id": "req_001",
  "message": "帮我规划一下学习路径",
  "metadata": {}
}
```

完成响应：

```json
{
  "request_id": "req_001",
  "session_id": "session_001",
  "run_id": "run_001",
  "state": "completed",
  "answer": "这是第一阶段的回答结果。",
  "need_user_input": false,
  "question": null
}
```

追问响应：

```json
{
  "request_id": "req_001",
  "session_id": "session_001",
  "run_id": "run_001",
  "state": "awaiting_user",
  "answer": null,
  "need_user_input": true,
  "question": "请补充必要信息。"
}
```

### 6.3 GET /session/{session_id}

用于调试会话快照。

第一阶段可以按 `user_id + agent_id + session_id` 查询。

### 6.4 POST /agent/run/stream

AG-UI 流式运行接口。

外部系统默认优先接入该接口。

第一阶段传输方式使用 SSE：

```text
Content-Type: text/event-stream
```

请求可以先复用 `/agent/run` 的请求结构。

后续如果外部系统直接传入 AG-UI 标准 Run 输入结构，由 `web/ag_ui.py` 做协议适配，再转换为内部 `RuntimeRequest`。

响应不再返回一次性 `AgentRunResponse`，而是按 AG-UI 事件顺序持续输出。

非流式 `/agent/run` 保留用于调试、脚本测试和后端单元验证。

## 七、状态流转

### 7.1 State

```text
new_request
ready_to_plan
missing_params
awaiting_user
completed
failed
```

### 7.2 Action

```text
answer_user
call_skill
call_agent
ask_user
```

### 7.3 State / Action 关系

| 当前 State | 允许 Action |
| --- | --- |
| `new_request` | `answer_user` / `call_skill` / `call_agent` / `ask_user` |
| `ready_to_plan` | `answer_user` / `call_skill` / `call_agent` / `ask_user` |
| `missing_params` | `call_skill` / `call_agent` / `ask_user` |
| `awaiting_user` | `answer_user` / `call_skill` / `call_agent` / `ask_user` |

### 7.4 终态

```text
completed
failed
```

终态不再继续自动执行 Action。

`completed` 和 `failed` 是单次 `agent_run` 的终态，不代表 `session_id` 被关闭。

同一 `session_id` 后续收到新的用户输入时，可以创建新的 `agent_run` 并从 `new_request` 开始执行。

## 八、Mock Executor 设计

第一阶段不实现真实 Skill，也不实现真实子 Agent。

`call_skill` 和 `call_agent` 只是 Loop 的 Action 类型。

第一阶段通过 mock executor 模拟这两类 Action 的外部返回，用于验证 Runtime、State、Harness 和多轮上下文承接。

mock executor 必须表现得像正常业务能力，但本质上仍是测试替身。

测试用例通过正常用户问题触发 `LLMLoopDecider` 选择不同 Action。

mock executor 根据真实 Action 参数返回结构化结果。

Harness 不模拟，必须由 LLM 判断执行结果是否完成当前阶段目标。

多轮测试下，mock executor 需要接收会话上下文。

ActionExecutor 调用 mock executor 时，除本轮用户输入外，还应传入：

```text
session_id
pending_action
missing_params
previous_action_result
conversation_summary
```

其中 `conversation_summary` 第一阶段可以由 Runtime 在内存中基于当前会话已知信息拼出，不要求接入第四阶段的完整对话记忆或向量召回。

### 8.1 mock skill executor

建议提供一个 mock skill executor：`content_extract_skill`。

它不是第一阶段真实 Skill 资产，也不要求存在 `skills/content_extract_skill/` 目录。

用途：

```text
从用户提供的文本中提取要点、标题、行动项或结构化摘要。
```

输入：

```json
{
  "content": "需要处理的文本",
  "extract_type": "key_points",
  "session_context": {
    "session_id": "session_001",
    "pending_action": {},
    "missing_params": [],
    "previous_action_result": null,
    "conversation_summary": ""
  }
}
```

输出：

```json
{
  "status": "success",
  "data": {},
  "summary": "content_extract_skill 提取完成",
  "missing_params": [],
  "error": null
}
```

返回规则由 mock 能力边界决定：

| 输入情况 | 返回 |
| --- | --- |
| 有明确 `content` | `success` |
| 本轮缺少 `content`，但 `session_context` 可补齐 | `success` |
| 本轮和 `session_context` 都缺少 `content` | `missing_params` |
| 内容不可读或明确超出 mock 能力 | `failed` |

mock skill executor 只负责返回测试数据，不负责判断结果是否可用。

多轮行为要求：

- 第一轮缺少原文时，返回 `missing_params: ["content"]`。
- 第二轮用户补充原文后，允许从本轮输入和 `pending_action` 中恢复原始提取意图。
- 如果上一轮是 `failed`，新一轮用户直接粘贴可读文本时，应按新的 Action 参数执行，不被上一轮失败阻塞。
- 输出中可以包含结构化要点、标题或行动项，供最终回答使用。

是否进入 `ready_to_plan / missing_params / failed` 由 LLM Harness 判断。

### 8.2 mock agent executor

建议提供一个 mock agent executor：`planning_worker`。

它不是第一阶段真实 worker Agent，也不要求存在 `agents/workers/planning_worker/` 目录。

用途：

```text
根据用户目标、背景和约束，产出学习路线、实施计划或推进方案。
```

输入：

```json
{
  "goal": "用户目标",
  "background": "背景信息",
  "constraints": {},
  "session_context": {
    "session_id": "session_001",
    "pending_action": {},
    "missing_params": [],
    "previous_action_result": null,
    "conversation_summary": ""
  }
}
```

输出：

```json
{
  "status": "success",
  "summary": "planning_worker 规划完成",
  "result": {},
  "missing_params": [],
  "error": null
}
```

返回规则由 mock 能力边界决定：

| 输入情况 | 返回 |
| --- | --- |
| 有明确 `goal` 和必要背景 | `success` |
| 本轮信息不完整，但 `session_context` 可补齐 | `success` |
| 本轮和 `session_context` 仍缺少目标、背景或约束 | `missing_params` |
| 请求超出 mock 能力或依赖不可用资源 | `failed` |

mock agent executor 只负责返回测试数据，不负责判断结果是否可用。

多轮行为要求：

- 第一轮规划诉求过泛时，返回缺失字段，例如 `goal`、`background`、`constraints`。
- 第二轮用户补充目标、时间、团队或约束后，允许结合 `pending_action` 继续完成规划。
- 如果第二轮仍然只补充了部分信息，继续返回更具体的 `missing_params`。
- 同一会话中，上一个任务 `completed` 后，用户提出关联规划任务时，应作为新的 Action 参数执行，但可以读取 `conversation_summary` 中的上一轮要点。

是否进入 `ready_to_plan / missing_params / failed` 由 LLM Harness 判断。

### 8.3 测试触发方式

测试用例使用自然用户问题触发不同链路。

例如：

| 用户问题类型 | 预期触发 |
| --- | --- |
| 简单说明类问题 | `answer_user` |
| 信息明显不足的问题 | `ask_user` |
| 文本处理类问题 | `call_skill` |
| 规划、拆解、分析类问题 | `call_agent` |

是否触发成功，不靠规则判断，最终由执行轨迹和 LLM Case Judge 判断。

## 九、中间步骤输出设计（AG-UI）

AG-UI 是本项目对外输出 Agent 运行过程的标准协议。

后续接入本框架的外部系统，默认按照 AG-UI 事件流消费 Agent 状态、中间步骤、工具调用和最终回答。

因此，第一阶段从架构上必须保证：

- 流式接口输出遵守 AG-UI 事件结构。
- Runtime 内部状态不直接裸露给前端。
- 所有中间步骤都通过 AG-UI Adapter 转换后输出。
- 非流式 `/agent/run` 可以继续返回简化 JSON，但不能作为外部系统默认协议。

### 9.1 协议边界

AG-UI 只负责 Agent 与用户界面之间的交互事件。

它不替代：

- Runtime Loop。
- Runtime Hook。
- 数据库持久化。
- MCP / 外部工具协议。
- 未来真实 Skill 或真实子 Agent 协议。

边界关系：

```text
RuntimeEngine
  ↓ internal runtime events
RuntimeHook ─────────────→ database
  ↓
AGUIEventAdapter ────────→ /agent/run/stream
  ↓
external UI / external system
```

### 9.2 第一阶段传输方式

第一阶段使用 SSE 输出 AG-UI 事件流。

每个 SSE `data` 都是一条 AG-UI JSON 事件。

示例：

```text
event: ag-ui
data: {"type":"RUN_STARTED","runId":"run_001","threadId":"session_001"}
```

后续如果需要 WebSocket，可以复用同一个 `AGUIEventAdapter`，只替换传输层。

### 9.3 事件映射

第一阶段内部事件到 AG-UI 事件的映射如下：

| Runtime 内部时机 | AG-UI 事件 | 说明 |
| --- | --- | --- |
| 创建 `agent_run` | `RUN_STARTED` | 一次 Agent Run 开始 |
| 创建或恢复 `session_state` | `STATE_SNAPSHOT` | 输出当前 UI 可见会话快照 |
| 进入一次 Loop step | `STEP_STARTED` | 输出当前 step 编号和 state 摘要 |
| LoopDecider 给出 Action | `STATE_DELTA` | 输出可展示的运行状态变化 |
| `answer_user` 开始生成回答 | `TEXT_MESSAGE_START` | assistant 消息开始 |
| `answer_user` 输出内容 | `TEXT_MESSAGE_CONTENT` | assistant 文本增量或整段文本 |
| `answer_user` 完成 | `TEXT_MESSAGE_END` | assistant 消息结束 |
| `ask_user` 生成追问 | `TEXT_MESSAGE_START / CONTENT / END` | 追问也作为 assistant 文本消息 |
| `call_skill` 开始 | `TOOL_CALL_START` | 名称为 `content_extract_skill` |
| `call_agent` 开始 | `TOOL_CALL_START` | 名称为 `planning_worker` |
| mock executor 参数准备完成 | `TOOL_CALL_ARGS` | 输出面向 UI 的脱敏参数摘要 |
| mock executor 返回 | `TOOL_CALL_END` | 输出结构化结果摘要 |
| Harness 评估完成 | `STATE_DELTA` | 输出 Harness 后的可展示状态 |
| `agent_run_step` 追加完成 | `STEP_FINISHED` | 一次 Loop step 完成 |
| 进入 `completed` | `RUN_FINISHED` | run 正常结束 |
| 进入 `failed` 或异常 | `RUN_ERROR` | run 异常结束 |

第一阶段不要求输出真实 token 级别流式文本。

如果模型调用暂时只能拿到整段回答，也必须按：

```text
TEXT_MESSAGE_START
TEXT_MESSAGE_CONTENT
TEXT_MESSAGE_END
```

的顺序输出。

### 9.4 事件字段约定

AG-UI 事件至少包含：

```text
type
runId
threadId
timestamp
```

其中：

| 字段 | 来源 |
| --- | --- |
| `runId` | `agent_run.id` |
| `threadId` | `session_id` |
| `messageId` | assistant 消息 ID 或 step 派生 ID |
| `toolCallId` | `tool_call.id` |
| `timestamp` | 事件创建时间 |

Runtime 内部字段如 `state_before`、`state_after`、`harness_feedback`、`action_detail` 不直接裸露。

如果前端需要展示过程，只输出面向 UI 的摘要字段，例如：

```json
{
  "type": "STATE_DELTA",
  "runId": "run_001",
  "threadId": "session_001",
  "delta": {
    "phase": "calling_tool",
    "label": "正在整理文本要点"
  }
}
```

### 9.5 中间步骤展示原则

中间步骤给 UI 看，但不是把内部实现泄漏给用户。

允许展示：

- 正在理解请求。
- 我准备干什么。
- 正在调用文本处理能力。
- 正在调用规划分析能力。
- 文本处理已完成，正在组织回答。
- 规划分析已完成，正在组织回答。
- 还缺少哪些信息，需要用户补充。
- 正在生成最终回答。
- 已完成。
- 无法继续完成，并给出面向用户的失败原因。

“我准备干什么”不展示内部步骤编号，也不展示 `Action` 名称。

它应该作为 assistant 可见文本流式返回，表达下一步安排，例如：

```text
我会先确认你要处理的是哪段文本，然后把它整理成几个关键要点。
```

或：

```text
我会先根据你的目标和时间约束拆出一版执行计划，再整理成可以直接推进的清单。
```

### 9.5.1 可见阶段建议

AG-UI Adapter 可以把内部状态映射成以下面向用户的可见阶段：

| 可见阶段 | 触发时机 | 推荐展示 |
| --- | --- | --- |
| `understanding` | 收到请求后 | 正在理解请求 |
| `planning_next` | LoopDecider 选出下一步后 | 我准备干什么 |
| `calling_text_skill` | `call_skill` 开始 | 正在整理文本 |
| `calling_planning_agent` | `call_agent` 开始 | 正在生成方案 |
| `tool_result_ready` | mock executor 返回后 | 已拿到处理结果，正在判断是否足够回答 |
| `need_more_info` | Harness 判断 `missing_params` 后 | 还需要补充信息 |
| `answering` | `answer_user` 开始 | 正在生成最终回答 |
| `completed` | run 正常结束 | 已完成 |
| `failed` | run 失败 | 无法继续完成 |

### 9.5.2 Action 展示映射

内部 Action 到 AG-UI 可见输出的建议映射：

| 内部 Action | AG-UI 事件 | 用户可见文案方向 |
| --- | --- | --- |
| `answer_user` | `TEXT_MESSAGE_START / CONTENT / END` | 直接生成最终回答 |
| `ask_user` | `TEXT_MESSAGE_START / CONTENT / END` | 说明缺少什么，并提出一个聚焦问题 |
| `call_skill` | `STATE_DELTA` + `TOOL_CALL_*` | 我会整理你提供的文本，提取要点或摘要 |
| `call_agent` | `STATE_DELTA` + `TOOL_CALL_*` | 我会拆解目标，生成方案或计划 |

`TOOL_CALL_*` 事件可以给外部系统消费，但普通聊天 UI 可以只展示更自然的 `STATE_DELTA.label`。

例如：

```json
{
  "type": "STATE_DELTA",
  "delta": {
    "phase": "planning_next",
    "label": "我会先整理你提供的文本，再提取出关键要点。"
  }
}
```

```json
{
  "type": "STATE_DELTA",
  "delta": {
    "phase": "calling_planning_agent",
    "label": "我正在根据你的目标和约束生成一版推进方案。"
  }
}
```

不允许展示：

- 原始 Prompt。
- `LLMLoopDecider` 原始输出。
- Harness 完整判断理由。
- 数据库表名。
- mock executor 是测试替身这件事。
- 内部错误堆栈。

### 9.6 与数据库记录的关系

AG-UI 事件是对外交互协议。

`agent_run_step` 和 `tool_call` 是内部持久化记录。

两者不要求字段一一相同，但必须能通过 `runId` 和 `threadId` 关联。

第一阶段建议：

- `agent_run_step` 继续完整记录内部状态变化。
- `tool_call` 继续记录 mock executor 输入输出。
- AG-UI 只输出可被 UI 消费的事件摘要。

### 9.7 第一阶段验收

AG-UI 中间步骤验收要求：

- `/agent/run/stream` 能输出 `RUN_STARTED`。
- 每次 Loop step 至少输出 `STEP_STARTED` 和 `STEP_FINISHED`。
- `answer_user` 和 `ask_user` 通过 `TEXT_MESSAGE_*` 输出。
- `call_skill` 和 `call_agent` 通过 `TOOL_CALL_START / TOOL_CALL_ARGS / TOOL_CALL_END` 输出。
- 会话状态变化可通过 `STATE_SNAPSHOT / STATE_DELTA` 输出。
- Run 正常结束输出 `RUN_FINISHED`。
- Run 异常结束输出 `RUN_ERROR`。
- 事件中不得暴露内部 Prompt、Harness 原文、异常堆栈和 mock 测试替身身份。

## 十、错误处理

第一阶段需要统一异常结构。

错误类型：

| 类型 | 场景 |
| --- | --- |
| `ValidationError` | API 参数错误 |
| `AgentLoadError` | Agent 文件缺失或解析失败 |
| `ActionParseError` | Loop 输出无法解析 |
| `ActionExecuteError` | Action 执行失败 |
| `HarnessError` | Harness 评估失败 |
| `StateError` | 非法状态流转 |
| `RuntimeHookError` | Hook 处理失败 |

Hook 错误不影响主链路，但必须记录。

非 Hook 错误根据严重程度进入 `failed` 或直接返回接口错误。

## 十三、验收标准

第一阶段完成时，需要满足：

- 服务可启动。
- `/health` 返回 ok。
- `/agent/run` 可创建新会话。
- `/agent/run` 可继续 `awaiting_user` 会话。
- `answer_user` 链路可完成。
- `ask_user` 链路可进入 `awaiting_user`。
- `call_skill` 成功后可回到 `ready_to_plan`。
- `call_skill` 缺参后可进入 `missing_params`。
- `call_skill` 失败后可进入 `failed`。
- `call_agent` 成功后可回到 `ready_to_plan`。
- `call_agent` 缺参后可进入 `missing_params`。
- `call_agent` 失败后可进入 `failed`。
- `call_skill` / `call_agent` 由 mock executor 模拟，不要求真实 Skill 或真实子 Agent。
- Harness 由大模型完成判断，不使用简单规则映射替代。
- 每个测试用例结束后，执行轨迹可交给大模型做完成度判定。
- `session_state` 只在边界状态更新。
- `agent_run_step` 每轮追加。
- `/agent/run/stream` 可按 AG-UI 输出中间步骤事件。
- Hook 失败不影响 `/agent/run` 主响应。
- 不需要真实记忆宫殿服务。
- 不需要 ES / OpenSearch。
- 不需要外部队列。
