# 第一阶段测试用例

## 概述

本文档定义第一阶段「Loop 全链路骨架」的测试用例。

第一阶段测试目标不是验证具体业务能力成熟度，而是验证：

- State / Action 流转完整。
- mock skill executor / mock agent executor 能在正常问题下触发分支。
- LLM Harness 能真实评估 mock executor 结果。
- `agent_run_step` 能记录完整状态轨迹。
- 每个用例结束后，模型能分析该用例是否完成预期链路。
- 正常用户输入下，Agent 的回答、追问和失败说明仍然自然可用。

## 一、测试原则

第一阶段测试分两层：

1. 状态机覆盖测试。
2. LLM 用例完成度判定。

mock executor 的职责：

- `content_extract_skill`：mock skill executor，用于模拟文本提取类外部返回。
- `planning_worker`：mock agent executor，用于模拟规划分析类外部返回。

大模型的职责：

- `LLM LoopDecider`：根据正常用户问题判断下一步 Action。
- `LLM Harness`：判断 mock executor 的一次执行结果是否可用。
- `LLM Case Judge`：判断整个测试用例是否完成预期链路。

禁止用简单规则替代 Harness 判断。

---

## 二、统一请求格式

所有测试用例通过 `/agent/run` 触发。

请求示例：

```json
{
  "user_id": "test_user",
  "agent_id": "main",
  "session_id": null,
  "request_id": "case_skill_success_001",
  "message": "帮我把这段项目说明整理成几个关键要点",
  "metadata": {}
}
```

测试必须保持：

- `message` 使用正常用户表达。
- 测试链路只通过用户消息、会话上下文和 Agent 可用能力触发。
- 返回给用户的内容不能暴露模拟能力、状态机等内部实现细节。

## 三、测试轨迹采集

每个测试用例执行后，需要采集：

```text
request
agent_run
agent_run_step[]
tool_call[]
final_response
session_state
```

其中 `agent_run_step[]` 至少包含：

```text
step_index
state_before
action
action_detail
execution_result
harness_feedback
state_after
```

## 四、LLM Harness 判定

Harness 判定发生在 Runtime 中。

它只评估 `call_skill` 和 `call_agent` 的执行结果。

Harness 输入：

```json
{
  "user_request": {},
  "state_before": "new_request",
  "action": "call_skill",
  "action_detail": {},
  "execution_result": {},
  "pending_action": {},
  "missing_params": [],
  "harness_instruction": "来自 harness.md 的评估规则"
}
```

Harness 输出：

```json
{
  "state": "ready_to_plan",
  "status": "success",
  "summary": "content_extract_skill 返回的信息可以支持继续规划",
  "missing_params": [],
  "reason": "结果完整且没有阻塞项"
}
```

允许输出的 `state`：

```text
ready_to_plan
missing_params
failed
```

## 五、LLM 用例完成度判定

用例完成度判定发生在测试结束后。

它判断整条测试链路是否覆盖预期状态机，并检查用户侧交互是否自然。

第一阶段的通过判定不要求所有自然用户请求都严格命中预设链路。
当用户请求明显缺少必要信息，或依赖当前无法访问的外部资料时，`LoopDecider` 可以直接选择 `ask_user` 并进入 `awaiting_user`。
这类链路属于正常产品行为，不应为了测试覆盖强行先调用 mock executor。

因此用例判定分为两类结果：

| 字段 | 含义 |
| --- | --- |
| `strict_path_pass` | 实际状态链路是否完全等于文档中的参考链路，用于观察状态覆盖情况 |
| `passed` | 用例是否通过，以用户交互自然、会话承接正确、补充信息后能回到正确能力链路并完成为准 |

也就是说，缺信息场景第一轮直接 `ask_user` 可以判定通过；但用户补充信息后，必须能进入对应的 `call_skill` 或 `call_agent` 链路，并最终完成。

判定输入：

```json
{
  "case_name": "skill_success",
  "expected_path": [
    "new_request",
    "call_skill",
    "ready_to_plan",
    "answer_user",
    "completed"
  ],
  "steps": [],
  "final_response": {},
  "tool_calls": [],
  "user_interaction": {
    "message": "帮我把这段项目说明整理成几个关键要点",
    "answer": "..."
  }
}
```

判定输出：

```json
{
  "passed": true,
  "strict_path_pass": true,
  "covered_states": ["new_request", "ready_to_plan", "completed"],
  "covered_actions": ["call_skill", "answer_user"],
  "interaction_quality": "pass",
  "missing_coverage": [],
  "reason": "执行轨迹符合预期，Harness 正确判断 mock executor 结果可用，最终回答没有暴露内部测试信息。"
}
```

判定原则：

- 测试脚本可以检查 JSON 是否可解析。
- 完成度结论由大模型输出。
- `strict_path_pass = false` 不等于用例失败。
- 当明显缺信息或外部资料不可访问时，直接追问用户是允许链路。
- 多轮用例的最终判定重点是补充信息后能否回到正确能力链路并完成。
- 如果模型判定失败，需要保留执行轨迹和失败原因。
- 模型判定不替代数据库约束和接口参数校验。
- 如果回答暴露模拟能力、内部 State、Action 或测试术语，用例应判定失败。

## 六、测试用例清单

测试用例分为两类：

- 单轮基础用例：保留简单用例，用于验证单个 State / Action 分支是否可达。
- 多轮链路用例：验证同一 `session_id` 下，Agent 能否正确承接上一轮状态，并在多次用户补充后完成任务。

### 6.1 单轮基础用例

单轮基础用例不追求复杂业务，只用于快速验证状态点位。

| 用例 | 用户输入 | 期望链路 | 关注点 |
| --- | --- | --- | --- |
| 直接说明能力 | 你是谁？你能帮我做什么？ | `new_request → answer_user → completed` | 不调用 mock executor，不进入 Harness |
| 信息不足时追问 | 帮我整理一下 | `new_request → ask_user → awaiting_user` | 保存 `pending_action`，返回自然追问 |
| 文本提取成功 | 帮我把这段项目说明整理成几个关键要点：第一阶段要先跑通 Agent Runtime... | `new_request → call_skill → ready_to_plan → answer_user → completed` | 调用 `content_extract_skill` mock executor，Harness 判断可继续 |
| 文本提取缺少内容 | 帮我提取一下重点 | `new_request → ask_user → awaiting_user`，或 `new_request → call_skill → missing_params → ask_user → awaiting_user` | 明显缺少原文时允许直接追问；如果已调用 mock executor，则由 Harness 判断缺参 |
| 文本提取失败 | 帮我读取这个已经损坏的附件并转换成结构化摘要 | `new_request → ask_user → awaiting_user`，或 `new_request → call_skill → failed` | 如果用户仍可补充可读内容，允许先追问；只有明确不可恢复时进入 `failed` |
| 规划分析成功 | 我有三个月时间，每周能学习 8 小时，想学 Python 做数据分析，帮我规划一条学习路线 | `new_request → call_agent → ready_to_plan → answer_user → completed` | 调用 `planning_worker` mock executor，最终回答可用 |
| 规划分析缺少背景 | 帮我做一个实施方案 | `new_request → ask_user → awaiting_user`，或 `new_request → call_agent → missing_params → ask_user → awaiting_user` | 明显缺少目标、背景或约束时允许直接追问；补充后必须能进入规划链路 |
| 规划分析失败 | 帮我基于一个无法访问的外部私有系统实时数据，分析这个项目下一步怎么推进 | `new_request → ask_user → awaiting_user`，或 `new_request → call_agent → failed` | 如果可以通过用户补充资料继续，允许先追问；只有明确不可恢复时进入 `failed` |

### 6.2 多轮链路记录要求

多轮用例必须按轮次记录，不只看最终结果。

每一轮都需要保存：

```text
turn_index
session_id
previous_session_state
user_message
expected_path
actual_agent_run_step[]
tool_call[]
harness_feedback
final_response
next_session_state
case_judge_result
```

判定重点：

- 第二轮以后必须复用上一轮返回的 `session_id`。
- 每轮都要检查 `state_before / action / state_after` 是否合理。
- 文档中的期望链路是参考链路，不是唯一允许链路。
- 明显缺信息的首轮可以直接 `ask_user → awaiting_user`，只要追问自然、会话状态可承接，即可判定该轮通过。
- 如果已经进入 `call_skill` 或 `call_agent`，则必须由 Harness 判断 `ready_to_plan / missing_params / failed`。
- 进入 `awaiting_user` 时应尽量保存 `pending_action`；如果没有经过 Harness，`missing_params` 可以为空，但追问内容必须表达清楚缺口。
- 用户补充信息后，Loop 需要先判断能否补齐参数，不是机械追问。
- 参数补齐后，可以重新 `call_skill` 或 `call_agent`，再由 LLM Harness 判断是否进入 `ready_to_plan`。
- 进入 `completed` 或 `failed` 后，本次 `agent_run` 终止；同一会话的新输入可以开启新的 `agent_run`。

### 6.3 多轮链路 A：文本提取缺参后补充完成

目标：

```text
用户先提出文本提取诉求，但没有提供原文；Agent 追问后，用户补充原文，Agent 完成提取。
```

轮次 1：

```json
{
  "session_id": null,
  "message": "帮我提取一下重点",
  "metadata": {}
}
```

期望链路：

```text
new_request → ask_user → awaiting_user
或
new_request → call_skill → missing_params → ask_user → awaiting_user
```

检查点：

- Loop 判断用户想做文本提取，但当前缺少原文。
- 可以直接追问用户补充原文。
- 如果调用了 `content_extract_skill` mock executor，则由 LLM Harness 判断缺少 `content`。
- `session_state.pending_action` 保存原始意图。
- 如果经过 Harness，`session_state.missing_params` 应包含 `content`。
- 用户侧返回自然追问，不暴露内部状态。

轮次 2：

```json
{
  "session_id": "轮次 1 返回的 session_id",
  "message": "原文是：第一阶段要先实现 Runtime、StateManager、LLM Harness 和 RuntimeHook，并用测试用例覆盖全部状态流转。",
  "metadata": {}
}
```

期望链路：

```text
awaiting_user → call_skill → ready_to_plan → answer_user → completed
```

检查点：

- Runtime 读取上一轮 `session_state`。
- Loop 判断用户补充内容已满足 `content`。
- 再次调用 `content_extract_skill` mock executor。
- LLM Harness 判断结果可用，进入 `ready_to_plan`。
- 最终回答完成提取任务。
- `pending_action` 和 `missing_params` 在完成后被清理或置为空。

### 6.4 多轮链路 B：规划缺少背景后补充完成

目标：

```text
用户先提出泛化规划诉求；Agent 追问后，用户补充目标、时间和约束，Agent 通过 mock agent executor 完成规划模拟。
```

轮次 1：

```json
{
  "session_id": null,
  "message": "帮我做一个实施方案",
  "metadata": {}
}
```

期望链路：

```text
new_request → ask_user → awaiting_user
或
new_request → call_agent → missing_params → ask_user → awaiting_user
```

检查点：

- Loop 判断用户想做规划，但当前缺少目标、背景或约束。
- 可以直接追问用户补充必要信息。
- 如果调用了 `planning_worker` mock executor，则由 LLM Harness 判断缺少目标、背景或约束。
- `session_state.pending_action` 保存规划意图。
- 用户侧追问需要补充的信息。

轮次 2：

```json
{
  "session_id": "轮次 1 返回的 session_id",
  "message": "这个方案面向第一阶段 Agent Runtime 落地，目标是两周内完成 API、Runtime、状态流转和测试用例，团队只有两个人。",
  "metadata": {}
}
```

期望链路：

```text
awaiting_user → call_agent → ready_to_plan → answer_user → completed
```

检查点：

- Loop 能把用户补充内容合并到上一轮规划意图。
- 调用 `planning_worker` mock executor。
- LLM Harness 判断 mock executor 结果可用于最终回答。
- 最终回答必须体现两周、两个人、第一阶段范围。

### 6.5 多轮链路 C：多次补充后才完成

目标：

```text
验证 Agent 在连续信息不足时，可以多次进入 awaiting_user，并保持同一个任务上下文。
```

轮次 1：

```json
{
  "session_id": null,
  "message": "帮我规划一下学习路线",
  "metadata": {}
}
```

期望链路：

```text
new_request → ask_user → awaiting_user
或
new_request → call_agent → missing_params → ask_user → awaiting_user
```

检查点：

- 保存规划意图。
- 追问学习目标或背景。

轮次 2：

```json
{
  "session_id": "轮次 1 返回的 session_id",
  "message": "我想学 Python",
  "metadata": {}
}
```

期望链路：

```text
awaiting_user → call_agent → missing_params → ask_user → awaiting_user
```

检查点：

- Loop 不丢失上一轮规划意图。
- LLM Harness 可以继续判断信息不足。
- `missing_params` 更新为更具体的缺口，例如学习周期、每周投入时间、当前基础。
- 第二次追问要比第一次更聚焦。

轮次 3：

```json
{
  "session_id": "轮次 1 返回的 session_id",
  "message": "我有三个月时间，每周能学习 8 小时，目前只会一点基础语法，目标是做数据分析。",
  "metadata": {}
}
```

期望链路：

```text
awaiting_user → call_agent → ready_to_plan → answer_user → completed
```

检查点：

- Runtime 仍然使用同一个 `session_id`。
- Loop 判断参数已补齐。
- 调用 `planning_worker` mock executor。
- 最终回答应覆盖目标、周期、每周时间、当前基础。

### 6.6 多轮链路 D：失败后同一会话重新发起新任务

目标：

```text
验证 `failed` 是单次 agent_run 的终态，不会让同一个 session 后续输入无法继续。
```

轮次 1：

```json
{
  "session_id": null,
  "message": "帮我读取这个已经损坏的附件并转换成结构化摘要",
  "metadata": {}
}
```

期望链路：

```text
new_request → ask_user → awaiting_user
或
new_request → call_skill → failed
```

检查点：

- 如果系统判断用户可以改为粘贴文本或提供可读资料，可以直接追问并进入 `awaiting_user`。
- 如果已经调用 `content_extract_skill` mock executor 且结果不可恢复，则由 LLM Harness 判断失败。
- 进入 `failed` 时，本次 `agent_run` 不继续自动执行 Action。

轮次 2：

```json
{
  "session_id": "轮次 1 返回的 session_id",
  "message": "我改成直接粘贴文本：第一阶段目标是跑通 Agent Runtime、LLM Harness、RuntimeHook 和状态流转测试，请帮我提取要点。",
  "metadata": {}
}
```

期望链路：

```text
new_request → call_skill → ready_to_plan → answer_user → completed
```

检查点：

- 新请求不被上一轮 `awaiting_user` 或 `failed` 阻塞。
- 创建新的 `agent_run`。
- 调用 `content_extract_skill` mock executor。
- 最终状态为 `completed`。

### 6.7 多轮链路 E：先提取，再基于结果继续规划

目标：

```text
验证同一会话中，用户可以在一个任务完成后继续提出关联任务。
```

轮次 1：

```json
{
  "session_id": null,
  "message": "帮我把这段内容提炼成要点：第一阶段要完成 FastAPI 接口、Agent Loader、LoopDecider、ActionExecutor、LLM Harness 和测试用例。",
  "metadata": {}
}
```

期望链路：

```text
new_request → call_skill → ready_to_plan → answer_user → completed
```

检查点：

- 调用 `content_extract_skill` mock executor。
- 最终回答给出结构化要点。

轮次 2：

```json
{
  "session_id": "轮次 1 返回的 session_id",
  "message": "基于刚才整理出的要点，帮我排一个一周内完成的实施计划。",
  "metadata": {}
}
```

期望链路：

```text
new_request → call_agent → ready_to_plan → answer_user → completed
```

检查点：

- 上一轮 `completed` 不阻止新任务。
- Loop 判断新任务更适合调用 `planning_worker` mock executor。
- 最终回答应围绕 FastAPI、Agent Loader、LoopDecider、ActionExecutor、LLM Harness、测试用例展开。
- 第一阶段如果还没有完整对话表，只要求同一 `session_id` 和上一轮摘要可被 Runtime 使用；更完整的语义召回放到第四阶段。

## 七、覆盖矩阵

| State / Action | 覆盖用例 |
| --- | --- |
| `new_request` | 全部单轮基础用例 / 多轮链路 D、E 的新任务轮次 |
| `ready_to_plan` | 文本提取成功 / 规划分析成功 / 多轮链路 A、B、C、D、E 的完成轮次 |
| `missing_params` | 已调用 mock executor 后发现缺参的用例 / 多轮链路 C 的中间轮次 / 专项覆盖用例 |
| `awaiting_user` | 信息不足时追问 / 多轮链路 A、B、C |
| `completed` | 直接说明能力 / 文本提取成功 / 规划分析成功 / 多轮链路 A、B、C、D、E 的完成轮次 |
| `failed` | 明确不可恢复的 mock executor 失败用例 / 专项覆盖用例 |
| `answer_user` | 直接说明能力 / 成功类用例 / 多轮链路完成轮次 |
| `ask_user` | 信息不足时追问 / 缺参类用例 / 多轮链路 A、B、C 的追问轮次 |
| `call_skill` | 文本提取类用例 / 多轮链路 A、D、E |
| `call_agent` | 规划分析类用例 / 多轮链路 B、C、E |
| `session_state` 承接 | 多轮链路 A、B、C |
| 终态后新 `agent_run` | 多轮链路 D、E |

## 八、验收要求

第一阶段测试通过条件：

- 所有测试用例均可执行。
- 每个用例都有 `agent_run`。
- 每个用例都有完整 `agent_run_step`。
- 多轮用例中，每一轮都必须有独立 `agent_run` 和完整 `agent_run_step`。
- 多轮用例必须复用同一个 `session_id`，并检查上一轮 `next_session_state` 与下一轮 `previous_session_state` 是否衔接。
- 发生 `call_skill` / `call_agent` 的用例必须有 `tool_call`。
- 明显缺信息的首轮直接 `ask_user` 属于允许链路，不要求必须产生 `tool_call`。
- 多轮用例只要补充信息后回到正确 `call_skill` / `call_agent` 链路并完成，即可判定通过。
- `strict_path_pass` 只用于覆盖观察，不作为 `passed` 的唯一依据。
- Harness 由大模型输出结构化判定。
- 每个用例结束后都有 LLM 用例完成度判定。
- 多轮用例需要在每一轮结束后做 LLM 判定，并在整条多轮链路结束后再做一次总体判定。
- 所有用例的 LLM 判定 `passed = true`。
- 所有用户侧回答、追问和失败说明都自然可读。
- 用户侧输出不暴露模拟能力、State、Action 等内部实现细节。
- Hook 失败测试不影响主响应。
