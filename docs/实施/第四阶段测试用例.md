# 第四阶段测试用例

## 概述

本文档定义第四阶段「会话上下文压缩闭环」测试用例。

第四阶段测试目标是验证：

- 当前 session 的原始 `conversation_turns` 可以按预算保留为 `recent_turns`。
- 短会话不触发摘要模型调用。
- 长会话接近 token 阈值后才触发 `conversation_summary` 压缩。
- 摘要只服务当前 `user_id + agent_id + session_id`。
- `recent_turns` 和 `event_dialogues` 可以按 `conversation_turn` 精确去重。
- 被压缩出 recent turns 的历史事件，可以通过 `event_dialogues` 找回原始细节。
- Prompt 中不会重复注入同一轮对话。
- 第四阶段不接入记忆宫殿、不接入 `memory_manager`。

## 一、测试原则

第四阶段测试必须保持：

- 每组测试使用独立 `user_id + session_id`。
- main 和 worker 的摘要按 `agent_id` 分开验证。
- 当前用户输入永远保留。
- `event_id / turn_id / run_id` 可以用于测试采集，但不能进入 Prompt 展示内容。
- 预算未达阈值时不允许调用摘要模型。
- 预算达到阈值时只压缩较早对话，最近对话必须保留原文。
- 默认历史上下文窗口按 `6000 tokens` 验证。
- 默认触发水位按 `4800 tokens` 验证。
- 默认压缩后目标水位按 `2100 tokens` 验证。
- 最近 `4` 个 `conversation_turns` 必须保护，即使它们本身超过 2100 tokens。
- 记忆宫殿相关字段不得成为第四阶段通过条件。

---

## 二、统一上下文结构检查

第四阶段 Prompt 上下文应包含：

```json
{
  "conversation_summary": "较早对话摘要",
  "recent_turns": [],
  "event_context": {
    "should_inject": true,
    "event_dialogues": []
  },
  "event_dialogues_deduped": [],
  "current_user_message": "当前用户输入"
}
```

Prompt 展示内容不应包含：

```text
event_id
turn_id
run_id
summary_batch_id
API Key
内部 Prompt
工具原始响应全量 JSON
```

## 三、conversation_turn 定义测试

### 3.1 conversation_turn 与 Loop step 区分

准备：

- 新 `user_id + agent_id + session_id`。
- 发起一次需要多步 Loop 的请求，例如路线和天气联合查询。
- Runtime 内部产生多个 Loop step。

检查点：

- 只记录一条用户可见的 `conversation_turn`。
- 这条 `conversation_turn` 包含用户原始输入和最终给用户的回答。
- Loop step 不被误当成多条 `conversation_turns`。
- `conversation_turn` 可以关联 `run_id` 和 `event_id`。

### 3.2 conversation_turn 作用域

准备：

- 同一 `session_id` 下 main 调用 `amap_worker`。
- main 和 worker 都产生用户可见或 worker 可见的一轮结果。

检查点：

- main 的 `conversation_turns` 作用域为 `user_id + main + session_id`。
- worker 的 `conversation_turns` 作用域为 `user_id + amap_worker + session_id`。
- main 和 worker 可以共享同一个 `event_id`。
- main 和 worker 不共享同一组 `conversation_turns`。

---

## 四、短会话不压缩

### 4.1 两轮普通对话

准备：

- 新 `user_id + session_id`。
- 连续完成两轮普通对话。
- 对话 token 估算低于 `4800 tokens`。

检查点：

- 不触发 `SessionContextCompressionRequested` 的模型总结调用。
- `session_state.conversation_summary` 为空或保持原值。
- 两轮对话都进入 `recent_turns`。
- Prompt 中不出现重复对话。

### 4.2 短会话事件追问

准备：

- 第一轮完成一个事件，例如路线和天气查询。
- 第二轮追问同一事件。
- 第一轮对话仍在 `recent_turns` 中。

检查点：

- `EventContextPreloader` 可以命中第一轮事件。
- `recent_turns` 已包含第一轮对话。
- `event_dialogues_deduped` 不再重复注入第一轮对话。
- Prompt 中第一轮对话只出现一次。
- 不触发会话摘要压缩。

---

## 五、长会话触发压缩

### 5.1 接近预算时压缩较早对话

准备：

- 新 `user_id + agent_id + session_id`。
- 构造多轮 `conversation_turns`，使未压缩 raw turns token 估算超过 `4800 tokens`。

期望链路：

```text
Run completed
  → 记录本轮 conversation_turn
  → Hook 检查 session raw conversation_turns token
  → token 超过 4800
  → SessionContextCompressionRequested
  → SessionContextCompressor
  → 更新 session_state.conversation_summary
```

检查点：

- 压缩前 `conversation_summary` 为空或旧摘要。
- 压缩后 `conversation_summary` 更新。
- 被压缩的是较早 `conversation_turns`。
- 压缩后未压缩 raw turns 尽量回到 `2100 tokens` 左右。
- 最近至少 `4` 轮原始对话仍保留。
- 如果最近 4 轮本身超过 `2100 tokens`，仍保护最近 4 轮。
- 当前用户输入不被压缩替代。

### 5.2 压缩摘要内容

准备：

- 待压缩 `conversation_turns` 中包含用户目标、已确认参数、已完成动作、待补充问题和当前结论。

检查点：

- 摘要保留用户当前目标。
- 摘要保留已确认参数。
- 摘要保留已完成动作。
- 摘要保留待补充问题。
- 摘要保留当前结论。
- 摘要不包含全量工具原始响应。
- 摘要不包含 API Key。
- 摘要不包含内部 Prompt。

### 5.3 压缩失败不影响主链路

准备：

- 模拟摘要模型调用失败。

检查点：

- 用户当前请求仍然完成或按正常 Loop 状态结束。
- Agent Run 不回滚。
- Hook 记录错误。
- `conversation_summary` 保持旧值。
- 下一轮仍可使用 recent turns。

---

## 六、摘要和 recent turns 拼装

### 6.1 摘要 + 最新对话同时进入 Prompt

准备：

- 当前 session 已经存在 `conversation_summary`。
- 当前 session 仍保留最近几轮 `recent_turns`。

检查点：

- Prompt 中包含 `conversation_summary`。
- Prompt 中包含 recent turns 原文。
- recent turns 不被摘要替代。
- 摘要不重复展开 recent turns 的原文。

### 6.2 最近对话优先级高于摘要

准备：

- 摘要中包含一个较早目标。
- 最近对话中用户明确修改目标。

检查点：

- Context Engineer 保留最近对话。
- LoopDecider 应以最近对话中的新目标为准。
- 摘要中的旧目标不能覆盖用户最新修改。

---

## 七、event_dialogues 去重

### 7.1 event_dialogues 与 recent_turns 重叠

准备：

- 第一轮完成路线查询事件。
- 第二轮立即追问。
- 第一轮仍在 `recent_turns` 中。
- `EventContextPreloader` 命中第一轮事件。

检查点：

- `recent_turns` 包含第一轮对话。
- `event_dialogues` 也包含第一轮对话。
- `ContextDeduplicator` 使用纯代码精确去重。
- 去重优先级为 `turn_id`、`run_id`、`sha1(agent_id + user + assistant)`。
- 不调用模型做去重判断。
- 不做语义相似度和模糊匹配。
- `ContextDeduplicator` 输出的 `event_dialogues_deduped` 为空或不包含该轮。
- Prompt 中第一轮对话只出现一次。
- 去重过程应为毫秒级。

### 7.2 event_dialogues 找回被压缩细节

准备：

- 第一轮完成路线查询事件。
- 后续产生大量对话，第一轮被压缩进 `conversation_summary`，不再在 `recent_turns` 中。
- 用户追问第一轮事件细节。

检查点：

- `conversation_summary` 提供早期任务概貌。
- `recent_turns` 不包含第一轮原始对话。
- `EventContextPreloader` 命中第一轮事件。
- `event_dialogues_deduped` 包含第一轮原始对话。
- Prompt 中可以看到第一轮原始对话。
- 没有重复注入。

### 7.3 多个事件候选只注入一个

准备：

- 当前 session 下存在多个已完成事件。
- 当前请求只与其中一个事件相关。

检查点：

- 相关性判断候选来自当前 session。
- 最多命中一个事件。
- 只注入命中事件内对话。
- 不注入其他事件对话。

---

## 八、作用域隔离

### 8.1 不跨 session 使用摘要

准备：

- 同一个 `user_id + agent_id` 下存在两个不同 `session_id`。
- session A 已经生成 `conversation_summary`。
- session B 发起新请求。

检查点：

- session B 不读取 session A 的 `conversation_summary`。
- session B 不读取 session A 的 `recent_turns`。
- session B 不读取 session A 的 event_dialogues。

### 8.2 不跨 agent 使用摘要

准备：

- 同一个 `user_id + session_id` 下 main 和 `amap_worker` 都有运行记录。
- main 有自己的 `conversation_summary`。
- worker 有自己的 `conversation_summary`。

检查点：

- main Prompt 使用 main 的摘要。
- worker Prompt 使用 worker 的摘要。
- main 和 worker 不共享同一条 session summary。

### 8.3 不跨用户使用摘要

准备：

- user A 和 user B 使用相同 agent 和相同业务问题。
- user A 已经生成摘要。

检查点：

- user B 不读取 user A 的摘要。
- user B 不读取 user A 的 event_dialogues。

---

## 九、worker 上下文测试

### 9.1 worker 保留自己的 recent turns

准备：

- main 多次调用 `amap_worker`。
- worker 在同一个 `session_id` 下产生多轮 run。

检查点：

- worker recent turns 来自 `agent_id = amap_worker`。
- worker 不把 main 的最终回答当成自己的 recent turns。
- Worker Task Package 仍可携带 main 的必要父上下文。

### 9.2 worker 事件对话去重

准备：

- worker 某轮 Skill 结果已经在自己的 recent turns 中。
- 同一事件被 `EventContextPreloader` 命中。

检查点：

- worker Prompt 不重复注入同一轮 worker 对话。
- worker 仍可根据已有上下文继续下一步 Skill 或回答。

---

## 十、非目标测试

第四阶段明确不测试：

- 记忆宫殿服务是否可用。
- `MemoryPalaceProvider` 是否接入。
- `memory_manager` worker 是否存在。
- 跨 session 长期记忆召回。
- 用户画像自动沉淀。
- 向量检索。

如果这些能力出现在第四阶段验收中，应视为范围漂移。

---

## 十一、具体执行用例

本节用例是可以直接用于手工测试、接口测试或自动化脚本的具体输入。

每个用例默认使用独立 `session_id`，除非用例明确要求复用。

### TC4-01：短会话两轮普通问答不压缩

测试目标：

- 验证短会话只写入 `conversation_turn`，不触发摘要压缩。
- 验证第二轮 Prompt 中包含第一轮 recent turn。

测试数据：

```text
user_id: user_tc4_01
agent_id: main
session_id: session_tc4_01
```

第 1 轮输入：

```text
你好，你能帮我做什么？
```

第 2 轮输入：

```text
我现在想先测试一下普通聊天链路，你简单回复一句就行。
```

预期结果：

- `conversation_turn` 中有 2 条 `user_id=user_tc4_01, agent_id=main, session_id=session_tc4_01` 的记录。
- 两条记录 `summarized_at` 都为空。
- `session_state.conversation_summary` 为空或保持旧值。
- 第 2 轮 Loop Prompt 的 `recent_turns` 中包含第 1 轮问答。
- 不发生会话摘要模型调用。

### TC4-02：短会话事件追问去重

测试目标：

- 验证事件对话命中后，如果同一轮已经在 `recent_turns` 中，不会重复注入。

测试数据：

```text
user_id: user_tc4_02
agent_id: main
session_id: session_tc4_02
```

第 1 轮输入：

```text
帮我查一下从杭州市西湖区文三路到杭州市上城区湖滨银泰的驾车路线，并顺便看一下目的地天气。
```

第 2 轮输入：

```text
那刚才这个目的地现在天气怎么样？
```

预期结果：

- 第 1 轮会产生 main 与 `amap_worker` 的运行记录。
- 第 2 轮 `EventContextPreloader` 可以命中第 1 轮事件。
- 第 2 轮 `recent_turns` 已包含第 1 轮 main 对话。
- 去重后 `event_context.event_dialogues` 不再重复包含第 1 轮 main 对话。
- Prompt 中第 1 轮 main 问答只出现一次。
- 不触发会话摘要压缩。

### TC4-03：长会话达到 4800 tokens 后触发压缩

测试目标：

- 验证未压缩 raw `conversation_turns` 达到约 4800 tokens 后触发摘要压缩。
- 验证最近 4 轮原始对话被保护。

测试数据：

```text
user_id: user_tc4_03
agent_id: main
session_id: session_tc4_03
```

连续输入 7 轮，每轮输入格式如下：

```text
第 N 轮：请记录这段业务背景，后续我会基于它继续提问。
背景内容：订单系统正在改造售后流程，涉及退款、退货、换货、人工审核、风控校验、客服备注、用户通知和财务对账。
请你只确认收到，并保留这些信息。
```

其中第 1 到第 7 轮都使用相同格式，但把 `第 N 轮` 替换为当前轮数，并在 `背景内容` 后追加一段约 800 到 1000 个中文字符的业务说明。

预期结果：

- 压缩触发前，`conversation_turn.summarized_at` 均为空。
- 达到阈值后，较早的 `conversation_turns` 被标记 `summarized_at` 和 `summary_batch_id`。
- 最近 4 条 `conversation_turns` 的 `summarized_at` 仍为空。
- `session_state.conversation_summary` 被更新。
- 压缩后未压缩 raw turns token 尽量回到约 2100 tokens。
- 如果最近 4 轮本身超过 2100 tokens，仍保留最近 4 轮，不强行压缩。

### TC4-04：摘要存在旧目标，recent turns 修改目标

测试目标：

- 验证 recent turns 优先级高于 `conversation_summary`。
- 验证旧摘要不能覆盖用户最新目标。

测试数据：

```text
user_id: user_tc4_04
agent_id: main
session_id: session_tc4_04
```

前置构造：

- 让 session 先产生一段 `conversation_summary`，摘要中包含：

```text
用户目标是设计一套售后退款流程，重点关注退款审核。
```

后续第 1 轮输入：

```text
刚才目标改一下，不做退款审核了，现在重点改成退货入库后的质检和重新上架流程。
```

后续第 2 轮输入：

```text
基于我最新说的目标，帮我列一个流程草案。
```

预期结果：

- 第 2 轮 Prompt 同时包含旧 `conversation_summary` 和最近两轮原始对话。
- LoopDecider / answer_user 应以 recent turns 中的“退货入库后的质检和重新上架流程”为准。
- 最终回答不应继续围绕“退款审核”展开。

### TC4-05：被压缩的历史事件通过 event_dialogues 找回细节

测试目标：

- 验证早期事件被压缩出 recent turns 后，用户追问时仍可通过事件对话找回原文细节。

测试数据：

```text
user_id: user_tc4_05
agent_id: main
session_id: session_tc4_05
```

第 1 轮输入：

```text
帮我查一下从杭州市西湖区文三路到杭州市上城区湖滨银泰的驾车路线，并看一下目的地天气。
```

第 2 到第 7 轮输入：

```text
我们先聊另一个业务背景：售后系统第 N 段资料如下。请确认收到。
资料：这里追加约 800 到 1000 个中文字符，让上下文超过压缩阈值。
```

第 8 轮输入：

```text
回到一开始那个路线问题，目的地是哪儿来着？天气结果也再结合一下说。
```

预期结果：

- 第 1 轮路线事件不再位于 main 的 `recent_turns` 中。
- `conversation_summary` 中包含早期路线事件概貌。
- 第 8 轮 `EventContextPreloader` 命中第 1 轮路线事件。
- 第 8 轮 Prompt 的 `event_context.event_dialogues` 包含第 1 轮路线原始问答。
- Prompt 不重复注入同一轮对话。
- 最终回答能正确提到目的地是“杭州市上城区湖滨银泰”，并结合天气结果回答。

### TC4-06：session 隔离

测试目标：

- 验证同一用户不同 session 之间不共享摘要和 recent turns。

测试数据：

```text
user_id: user_tc4_06
agent_id: main
session_id_A: session_tc4_06_a
session_id_B: session_tc4_06_b
```

session A 输入：

```text
请记住当前这个会话里，我们在讨论售后退款审核流程。
```

session B 输入：

```text
我们现在讨论一个新问题：门店库存盘点流程怎么设计？
```

预期结果：

- session B 的 Prompt 不包含 session A 的 `conversation_summary`。
- session B 的 Prompt 不包含 session A 的 `recent_turns`。
- session B 的回答不应主动提到“售后退款审核流程”。

### TC4-07：worker 拥有自己的 recent turns

测试目标：

- 验证 main 和 worker 在同一 session 下分别维护自己的 `conversation_turns`。

测试数据：

```text
user_id: user_tc4_07
main_agent_id: main
worker_agent_id: amap_worker
session_id: session_tc4_07
```

第 1 轮输入：

```text
帮我查一下杭州市上城区湖滨银泰今天的天气。
```

第 2 轮输入：

```text
再查一下从杭州市西湖区文三路开车过去要多久。
```

预期结果：

- `conversation_turn` 中存在 `agent_id=main` 的记录。
- `conversation_turn` 中存在 `agent_id=amap_worker` 的记录。
- main 的 `recent_turns` 只读取 `agent_id=main` 的记录。
- worker 的 `recent_turns` 只读取 `agent_id=amap_worker` 的记录。
- worker 不把 main 的最终回答当作自己的 recent turns。

### TC4-08：压缩失败不影响当前回答

测试目标：

- 验证摘要 Hook 失败不会阻断用户当前请求。

测试数据：

```text
user_id: user_tc4_08
agent_id: main
session_id: session_tc4_08
```

构造方式：

- 先通过多轮长文本让 raw turns 达到压缩阈值。
- 临时让摘要模型调用失败，例如使用无效模型地址或在测试中 mock `SessionContextCompressor.compress` 抛异常。

最后一轮输入：

```text
请基于我刚才提供的资料，简单总结一句当前讨论主题。
```

预期结果：

- 用户当前请求仍然返回正常回答，或按 Loop 正常进入 `completed / awaiting_user / failed` 中的合理状态。
- `AgentRun` 不回滚。
- `conversation_turn` 已记录当前轮。
- Hook 记录错误日志。
- `session_state.conversation_summary` 保持失败前的旧值。

## 十三、验收要求

第四阶段测试通过条件：

- 预算未达阈值时不调用摘要模型。
- 未压缩 raw `conversation_turns` 达到约 `4800 tokens` 时触发摘要压缩。
- 压缩后 `session_state.conversation_summary` 更新。
- 压缩后未压缩 raw turns 尽量回到约 `2100 tokens`。
- 最近 `4` 轮原始对话必须保护，即使超过 `2100 tokens`。
- 当前用户输入永远保留。
- Prompt 中 summary、recent turns、event dialogues 分层清晰。
- recent turns 与 event dialogues 通过纯代码精确去重，不重复注入。
- 被压缩出 recent turns 的历史事件可以通过 event dialogues 找回原始细节。
- 摘要按 `user_id + agent_id + session_id` 隔离。
- worker 拥有自己的会话上下文。
- 不接入记忆宫殿。
- 不接入 `memory_manager`。
- 不做跨用户或跨 session 召回。
