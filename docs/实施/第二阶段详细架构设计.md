# 第二阶段详细架构设计

## 概述

本文档描述第二阶段「Skill 闭环」的实施架构。

第一阶段已经跑通了 Agent Runtime 的最小闭环：

```text
User Request
  ↓
RuntimeEngine
  ↓
LoopDecider
  ↓
ActionExecutor
  ↓
mock skill / mock worker
  ↓
HarnessEvaluator
  ↓
Answer / Ask User
```

第二阶段的目标是把 `call_skill` 从 mock executor 替换为真实 Skill 运行机制。

第二阶段完成后，main agent 可以通过目录扫描得到可用 Skill，由 LoopDecider 根据 Skill 说明选择 `call_skill`，Runtime 校验输入并执行 Skill，Skill 结果继续进入 Harness，再回到 Loop。

第三阶段 worker 闭环会直接复用这一套 Skill 能力。

参考文档：

| 文档 | 说明 |
| --- | --- |
| `../技术方案/脚手架设计.md` | Agent 和 Skill 目录结构 |
| `../技术方案/技术方案.md` | 总体技术方案和阶段路线 |

## 一、第二阶段范围

### 要实现

第二阶段需要实现：

- `agents/main/skills/` 目录扫描。
- `SKILL.md` 读取。
- `schema.json` 读取和结构校验。
- SkillDefinition 构建。
- SkillRegistry 加载 main agent 可用 Skill。
- SkillCatalog 注入 LoopDecider Prompt。
- `call_skill` 使用真实 `skill_id` 和 `input`。
- Skill 输入参数 JSON Schema 校验。
- Skill Executor 执行。
- SkillAdapter 支持 `api / chain / code` 三类执行方式。
- Skill 输出结构校验。
- Skill 结果进入 LLM Harness。
- `tool_call` 保存真实 Skill 调用记录。
- AG-UI 输出真实 Skill 名称、参数摘要和结果摘要。
- 至少落地一个真实示例 Skill，用于替换第一阶段的 `content_extract_skill` mock executor。

### 不实现

第二阶段暂不实现：

- 真实 worker agent。
- worker 自己的 Skill 扫描。
- 跨 Agent 共享 Skill 市场。
- 外部插件市场。
- 任意未审计代码的动态加载。
- Skill 权限系统。
- Skill 沙箱隔离。
- Skill 长耗时异步任务队列。
- 记忆闭环。
- 记忆宫殿管理接口。
- conversation_record 向量召回。

第二阶段只解决 main agent 调用真实 Skill 的闭环。

其中 Skill 内部执行方式需要兼容三类：

```text
api    # 调用外部 HTTP / RPC 接口
chain  # 编排一条内部链路
code   # 执行 Runtime 白名单注册过的本地代码
```

## 二、目标链路

第二阶段需要验证以下链路：

```text
new_request → call_skill → ready_to_plan → answer_user → completed
new_request → call_skill → missing_params → ask_user → awaiting_user
new_request → call_skill → failed

awaiting_user → call_skill → ready_to_plan → answer_user → completed
missing_params → call_skill → ready_to_plan → answer_user → completed
ready_to_plan → call_skill → ready_to_plan → answer_user → completed
```

说明：

- `call_skill` 不再走第一阶段 mock skill executor。
- `call_agent` 在第二阶段仍可保留第一阶段 mock worker executor。
- Harness 仍然真实使用大模型判断 Skill 结果是否可继续。
- 明显缺信息的首轮仍然允许直接 `ask_user → awaiting_user`。
- 用户补充信息后，必须能回到对应 `call_skill` 链路并完成。

## 四、Skill 资产设计

### 4.1 Skill 最小结构

每个 Skill 目录保持最小资产结构：

```text
skills/
└── content_extract/
    ├── SKILL.md
    ├── schema.json
    ├── references/
    └── assets/
```

第二阶段不要求在 Skill 目录中放 Python 执行代码。

推荐做法是：

```text
Skill 资产目录：描述 Skill 是什么、怎么用、输入输出是什么
SkillAdapter：决定 Skill 通过 api、chain 还是 code 执行
Runtime builtin executor：承载已审计的本地 code Skill
```

这样可以避免 Runtime 直接执行 Agent 资产目录下的任意代码。

第二阶段需要支持三类 Skill：

```text
API Skill
Chain Skill
Code Skill
```

其中：

| 类型 | 说明 |
| --- | --- |
| `api` | 调用外部 HTTP / RPC 接口 |
| `chain` | 编排一条内部链路，例如 LLM → 解析 → 校验 → 调用接口 |
| `code` | 执行 Runtime 白名单注册过的本地 executor |
 
三类 Skill 都必须遵守同一套输入输出结构。

### 4.2 SKILL.md

`SKILL.md` 面向 Agent 和 Runtime。

建议包含：

```markdown
# content_extract

## 用途

从用户提供的文本中提取摘要、关键要点、结构化条目。

## 适用场景

- 用户提供了一段明确文本。
- 用户要求总结、提炼、归纳、结构化。

## 不适用场景

- 用户没有提供原始文本。
- 用户要求读取不可访问文件。
- 用户要求基于外部实时数据分析。

## 调用注意事项

- 必须提供 content。
- 可选提供 extract_mode。
- 输出必须是结构化结果，不能直接面向用户。
```

### 4.3 schema.json

`schema.json` 约束输入和输出。

第二阶段强制校验 `skill_id / executor / input_schema / output_schema` 等基础字段。

`executor` 用于声明 Skill 执行方式。

第二阶段可以忽略未知的顶层扩展字段，但不能忽略 `executor`、`input_schema` 和 `output_schema` 内部校验。

建议结构：

```json
{
  "skill_id": "content_extract",
  "name": "文本提取",
  "description": "从文本中提取摘要、关键要点或结构化条目。",
  "executor": {
    "type": "code",
    "name": "content_extract"
  },
  "input_schema": {
    "type": "object",
    "properties": {
      "content": {
        "type": "string",
        "description": "需要处理的原始文本"
      },
      "extract_mode": {
        "type": "string",
        "enum": ["key_points", "summary", "outline"],
        "description": "提取方式"
      }
    },
    "required": ["content"],
    "additionalProperties": false
  },
  "output_schema": {
    "type": "object",
    "properties": {
      "status": {
        "type": "string",
        "enum": ["success", "partial_success", "missing_params", "failed"]
      },
      "data": {
        "type": "object"
      },
      "summary": {
        "type": "string"
      },
      "missing_params": {
        "type": "array",
        "items": {
          "type": "string"
        }
      },
      "error": {
        "type": ["string", "null"]
      }
    },
    "required": ["status", "data", "summary", "missing_params", "error"],
    "additionalProperties": false
  }
}
```

第二阶段需要使用 JSON Schema 校验输入和输出。

建议新增依赖：

```text
jsonschema==4.23.0
```

## 五、核心模块

### 5.1 SkillDefinition

`SkillDefinition` 是 Runtime 内部的 Skill 元数据结构。

建议字段：

```python
class SkillDefinition:
    skill_id: str
    name: str
    description: str
    skill_dir: Path
    instruction: str
    input_schema: dict
    output_schema: dict
    references_dir: Path | None
    assets_dir: Path | None
```

其中：

- `skill_id` 来自目录名或 `schema.json.skill_id`。
- 二者不一致时应报错。
- `instruction` 来自 `SKILL.md`。
- `input_schema` 和 `output_schema` 来自 `schema.json`。

### 5.2 SkillLoader

`SkillLoader` 负责扫描当前 Agent 的 `skills/` 目录。

输入：

```text
agents/main/skills/
```

输出：

```text
list[SkillDefinition]
```

加载规则：

- 只扫描一级子目录。
- 忽略隐藏目录，例如 `.DS_Store`。
- 每个 Skill 必须包含 `SKILL.md` 和 `schema.json`。
- `schema.json` 必须可解析为 JSON 对象。
- `skill_id` 必须和目录名一致。
- Skill 说明为空时不允许加载。
- schema 缺少 `input_schema` 或 `output_schema` 时不允许加载。

加载失败属于 Agent 资产错误。

第二阶段可以先复用 `AgentLoadError`，如果后续 Skill 错误类型变多，再单独新增 `SkillLoadError`。

### 5.3 SkillRegistry

`SkillRegistry` 管理当前 Agent 可用 Skill。

职责：

- 保存 `skill_id → SkillDefinition` 映射。
- 保存 adapter 类型和已注册 code executor 映射。
- 提供 `get(skill_id)`。
- 提供 `list_for_prompt()`。
- 提供 `validate_skill_exists(skill_id)`。

`list_for_prompt()` 输出给 LoopDecider，不应暴露完整 schema。

建议输出：

```json
[
  {
    "skill_id": "content_extract",
    "name": "文本提取",
    "description": "从文本中提取摘要、关键要点或结构化条目。",
    "required_input": ["content"],
    "optional_input": ["extract_mode"]
  }
]
```

### 5.4 SkillAdapter

第二阶段需要定义统一 SkillAdapter 抽象：

```python
class BaseSkillAdapter:
    adapter_type: str

    async def execute(self, skill, payload: dict, context) -> dict:
        raise NotImplementedError
```

返回结构必须符合统一结果：

```json
{
  "status": "success",
  "data": {},
  "summary": "执行摘要",
  "missing_params": [],
  "error": null
}
```

第二阶段至少实现三类 Adapter：

| Adapter | executor.type | 职责 |
| --- | --- | --- |
| `ApiSkillAdapter` | `api` | 调用外部 HTTP / RPC 接口，并转换为统一 SkillResult |
| `ChainSkillAdapter` | `chain` | 编排一条内部链路，并产出统一 SkillResult |
| `CodeSkillAdapter` | `code` | 执行 Runtime 白名单注册过的本地 executor |

#### ApiSkillAdapter

`ApiSkillAdapter` 用于调用外部接口。

`schema.json.executor` 示例：

```json
{
  "type": "api",
  "endpoint_env": "TEXT_EXTRACT_API_URL",
  "method": "POST",
  "timeout_ms": 10000
}
```

规则：

- 接口地址从 `.env` 读取，不写死在 Prompt。
- 支持 `GET / POST` 即可。
- 超时必须可配置。
- 接口返回必须转换成统一 SkillResult。
- HTTP 错误、超时、返回结构错误都不能直接暴露给用户。

#### ChainSkillAdapter

`ChainSkillAdapter` 用于编排一条内部链路。

`schema.json.executor` 示例：

```json
{
  "type": "chain",
  "chain_id": "extract_then_normalize"
}
```

规则：

- `chain_id` 必须映射到 Runtime 内部已注册链路。
- 链路步骤不能由用户输入直接动态生成。
- 每个步骤都必须产生结构化中间结果。
- 最终必须转换成统一 SkillResult。
- 第二阶段只需要支持项目内注册的最小链路，不实现复杂 DAG。

#### CodeSkillAdapter

`CodeSkillAdapter` 用于执行本地已注册代码。

`schema.json.executor` 示例：

```json
{
  "type": "code",
  "name": "content_extract"
}
```

规则：

- 只执行 Runtime 白名单注册过的 executor。
- 不从 Skill 目录动态 import 任意代码。
- executor 抛出异常时转成 Runtime 系统错误或 Skill failed 结果。

### 5.5 SkillExecutor

`SkillExecutor` 是 ActionExecutor 调用真实 Skill 的入口。

执行流程：

```text
resolve skill_id
  ↓
load SkillDefinition from SkillRegistry
  ↓
merge action input and session_context if needed
  ↓
validate input_schema
  ↓
resolve SkillAdapter by executor.type
  ↓
execute adapter
  ↓
validate output_schema
  ↓
convert to ActionResult
```

输入校验分两种情况：

| 情况 | 处理 |
| --- | --- |
| 用户缺少业务参数 | 返回 `ActionResult(status="missing_params")`，交给 Harness 判断 |
| Runtime / schema / executor 结构错误 | 抛出业务异常，进入错误处理 |

第二阶段不能为了提升效果在 SkillExecutor 里写死用户意图路由。

用户意图仍然由 LoopDecider 判断。

SkillExecutor 只根据 `skill_id` 找到 SkillDefinition，再根据 `executor.type` 选择 Adapter。

### 5.6 ActionExecutor

第二阶段 `ActionExecutor._call_skill()` 不再调用 `ContentExtractMockExecutor`。

新的职责：

```text
ActionExecutor
  ↓
SkillExecutor.execute(skill_id, input, context)
  ↓
ActionResult
```

`ActionDecision.action_detail` 建议结构：

```json
{
  "skill_id": "content_extract",
  "input": {
    "content": "需要提取的文本",
    "extract_mode": "key_points"
  }
}
```

兼容第一阶段字段：

```json
{
  "name": "content_extract_skill",
  "input": {}
}
```

第二阶段可以在解析时兼容 `name`，但内部统一转换为 `skill_id`。

### 5.7 LLMLoopDecider

第二阶段 LoopDecider 不再写死：

```text
content_extract_skill 用于文本提取
```

而是从 Context 中读取 SkillCatalog。

Prompt 输入需要增加：

```json
{
  "available_skills": [
    {
      "skill_id": "content_extract",
      "name": "文本提取",
      "description": "从文本中提取摘要、关键要点或结构化条目。",
      "required_input": ["content"],
      "optional_input": ["extract_mode"]
    }
  ]
}
```

输出要求：

```json
{
  "action": "call_skill",
  "action_detail": {
    "skill_id": "content_extract",
    "input": {
      "content": "..."
    }
  },
  "reason": "用户要求提取文本要点，且已提供原文。"
}
```

如果信息明显不足，可以继续选择 `ask_user`。

不能为了覆盖测试强迫先调用 Skill。

### 5.8 ContextAssembler

`ContextAssembler` 需要把 SkillCatalog 注入 RuntimeContext。

建议新增字段：

```python
class RuntimeContext:
    skill_catalog: list[dict]
```

Context 来源：

```text
AgentDefinition
SessionState
Previous action result
Harness feedback
SkillRegistry.list_for_prompt()
```

第二阶段不把 Skill references 全量塞进 Prompt。

如果某个 Skill 需要参考资料，由 SkillExecutor 在执行时读取 `references/` 或 `assets/`。

### 5.9 HarnessEvaluator

Harness 继续评估 `call_skill` 结果。

第二阶段需要在 Harness 输入里补充 Skill 元信息：

```json
{
  "skill": {
    "skill_id": "content_extract",
    "name": "文本提取",
    "description": "从文本中提取摘要、关键要点或结构化条目。"
  },
  "execution_result": {}
}
```

Harness 仍然只允许输出：

```text
ready_to_plan
missing_params
failed
```

Harness 不直接返回用户。

`answer_user` 仍然由 Loop 再次判断后生成。

### 5.10 AG-UI Adapter

第二阶段 AG-UI 的 `TOOL_CALL_*` 事件应使用真实 Skill 信息。

示例：

```json
{
  "type": "TOOL_CALL_START",
  "toolName": "content_extract",
  "label": "正在整理文本"
}
```

`TOOL_CALL_ARGS` 只输出脱敏摘要。

不输出完整大文本、不输出原始 Prompt、不输出 schema 全量内容。

`TOOL_CALL_END` 输出：

```json
{
  "status": "success",
  "summary": "已提取 5 个关键要点",
  "missingParams": []
}
```

## 六、数据落地

第二阶段原则上不新增数据库表。

继续使用第一阶段表：

```text
agent_run
agent_run_step
tool_call
```

### tool_call

`tool_call` 记录真实 Skill 调用。

建议字段使用方式：

| 字段 | 第二阶段写入 |
| --- | --- |
| `action` | `call_skill` |
| `tool_name` | `skill_id` |
| `input_payload` | 已校验后的 Skill 输入和必要 session_context 摘要 |
| `output_payload` | Skill 标准结果 |
| `status` | Skill result status |

第二阶段不新增 `skill_call` 表。

原因：

- `tool_call` 已经能表达 Skill / worker 调用记录。
- 第三阶段 worker 也能继续复用。
- 等调用量上来后，再考虑 ES / OpenSearch 等日志中间件。

## 七、配置

第二阶段新增 Skill 相关环境变量。

```text
AMAP_WEB_SERVICE_KEY=***
SKILL_API_TIMEOUT_MS=10000
```

配置原则：

- API Key 只写在 `.env` 或服务端配置中。
- API Key 不写入 `SKILL.md`。
- API Key 不写入 `schema.json`。
- API Key 不注入 Prompt。
- API Key 不出现在 AG-UI 事件。
- API Key 不出现在用户回答。

高德 API Skill 通过 `schema.json.executor.auth.env` 声明读取哪个环境变量。

## 八、错误处理

第二阶段建议区分三类错误。

### 8.1 用户输入缺参

例如用户说：

```text
帮我提取重点
```

但没有提供原文。

允许链路：

```text
new_request → ask_user → awaiting_user
```

或：

```text
new_request → call_skill → missing_params → ask_user → awaiting_user
```

这不是系统异常。

### 8.2 Skill 业务失败

例如用户提供的内容不可读，或 Skill 能力明确无法处理。

Skill 返回：

```json
{
  "status": "failed",
  "data": {},
  "summary": "内容不可读，无法提取",
  "missing_params": [],
  "error": "unreadable_content"
}
```

结果进入 Harness，由 Harness 判断是否进入 `failed`。

### 8.3 Runtime 系统错误

例如：

- Skill 不存在。
- `schema.json` 非法。
- 输入输出结构不符合 schema。
- executor 抛出未捕获异常。

这类错误属于系统错误。

处理原则：

- 写入 `agent_run.error_message`。
- 写入日志。
- 对 AG-UI 输出 `RUN_ERROR`。
- 不伪装成用户缺参。

### 9.6 Chain Skill：amap_route_weather_plan

用途：

- 用户只提供起点和终点地址时，自动完成地址解析、路径规划、目的地天气查询和出行建议生成。

执行方式：

```json
{
  "executor": {
    "type": "chain",
    "chain_id": "amap_route_weather_plan"
  }
}
```

输入：

```json
{
  "origin_address": "杭州市西湖区文三路",
  "destination_address": "杭州市上城区湖滨银泰",
  "city": "杭州",
  "route_extensions": "base",
  "weather_extensions": "base"
}
```

链路步骤：

```text
1. amap_geocode(origin_address, city)
2. amap_geocode(destination_address, city)
3. amap_direction_driving(origin.location, destination.location)
4. amap_weather(destination.adcode)
5. travel_briefing_formatter(route, weather)
```

输出：

```json
{
  "status": "success",
  "data": {
    "origin": {},
    "destination": {},
    "route": {},
    "weather": {},
    "briefing": {},
    "steps": []
  },
  "summary": "已完成路线、天气和出行建议组合",
  "missing_params": [],
  "error": null
}
```

链路原则：

- 每一步都必须返回结构化中间结果。
- 任一步失败时，ChainSkillAdapter 需要返回可解释的 `failed` 或 `partial_success`。
- Chain Skill 结果仍然进入 Harness，不直接返回用户。
- Chain Skill 不应让模型动态生成任意步骤，本阶段只执行 Runtime 注册的固定链路。

## 十、与第一阶段的关系

第二阶段保留第一阶段能力：

- `/agent/run`
- `/agent/run/stream`
- AG-UI 中间步骤输出。
- LLM Harness。
- Runtime Hook。
- `agent_run_step`。
- `tool_call`。
- mock worker executor。

第二阶段替换第一阶段能力：

| 第一阶段 | 第二阶段 |
| --- | --- |
| `ContentExtractMockExecutor` | `SkillExecutor + content_extract Skill` |
| LoopDecider 写死 mock skill 说明 | LoopDecider 读取 SkillCatalog |
| AG-UI toolName 固定 `content_extract_skill` | AG-UI toolName 使用真实 `skill_id` |
| tool_call 记录 mock executor | tool_call 记录真实 Skill |

第二阶段不替换：

```text
planning_worker mock executor
```

真实 worker 在第三阶段处理。

- 外部任务队列。
